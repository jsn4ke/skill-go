# MMO 法术冷却与充能管理系统：完整设计

> 一份面向游戏服务端工程师的法术冷却与充能管理系统设计参考。本文基于对 TrinityCore（World of Warcraft 私有服务器）SpellHistory 系统的深度分析，以纯概念方式描述其架构，使读者能够在自己的项目中复刻类似系统。本文是施法状态机、效果管线、目标选择、Aura 架构、脚本扩展文章的姊妹篇。

## 1 引言

### 1.1 冷却系统的定位

每个 MMO 都面临一个基础问题：**技能不能无限使用**。如果玩家可以不受限制地施放最强技能，游戏的所有策略深度都将消失。

冷却系统就是解决这个问题的核心机制。它在法术施放成功后，为该法术设置一段"禁用期"，在禁用期内该法术不可再次使用。同时，冷却系统还管理着更复杂的时间状态：

```
┌─────────────────────────────────────────────────────────────────┐
│                     冷却管理系统的职责                            │
│                                                                 │
│   ┌──────────────────┐  法术施放后，记录"这个法术多久后可以再用"    │
│   │  法术冷却管理     │                                          │
│   ├──────────────────┤  同一分类的多个法术共享冷却                │
│   │  分类冷却         │  （如所有治疗药水共享 30 秒冷却）          │
│   ├──────────────────┤  多次使用后逐个恢复                        │
│   │  充能系统         │  （如技能有 3 次充能，用完一个恢复一个）    │
│   ├──────────────────┤  几乎所有法术都受其约束的短冷却             │
│   │  全局冷却 (GCD)   │                                          │
│   └──────────────────┘  被打断后，整个法术系被临时封锁             │
│   │  法术系封锁       │  （如被打断后无法施放任何火焰法术）         │
│   └──────────────────┘                                          │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 四个子系统的协作关系

这四个子系统不是孤立的，它们统一由一个核心类管理——每个游戏单位（Unit）持有一个冷却管理器实例。法术施放成功后，法术系统调用冷却管理器的入口方法，由冷却管理器统一决定如何处理：

```
法术施放成功
      │
      ▼
冷却管理器.HandleCooldowns(法术信息, 物品ID, 法术对象)
      │
      ├── 消费充能（如果有充能分类）
      │
      ├── 检查是否忽略冷却 ──→ 是 ──→ 结束
      │
      ├── 检查是否有忽略冷却光环 ──→ 是 ──→ 结束
      │
      ├── 药水/事件触发冷却？ ──→ 是 ──→ OnHold 状态
      │
      ├── 被动法术？ ──→ 是 ──→ 结束
      │
      └── 正常启动冷却（经修正器链计算后注册）
```

### 1.3 与法术系统的关系

冷却系统在法术处理流程中扮演"守门人"的角色。在法术施放之前，状态机会调用冷却系统的可用性检查；在法术施放成功之后，状态机会调用冷却系统的冷却注册入口：

```
法术施放时间线
═══════════════════════════════════════════════════════════▶

          可用性检查                        冷却注册
    ┌────────────────┐              ┌────────────────┐
    │ 冷却系统.IsReady│              │ 冷却系统.      │
    │                │              │ HandleCooldowns│
    │ 检查:          │              │                │
    │ - 法术系封锁？  │              │ 处理:          │
    │ - 冷却中？      │              │ - 消费充能     │
    │ - 有充能？      │              │ - 启动冷却     │
    └────────────────┘              └────────────────┘
```

- 希望在自己的项目中实现冷却/充能系统的服务端工程师

## 2 核心数据结构

### 2.1 冷却条目

冷却条目是冷却系统最基本的存储单元，记录一个法术从"施放完成"到"可以再次使用"的时间窗口：

```
冷却条目
├── 法术 ID            — 哪个法术处于冷却中
├── 冷却结束时间        — 法术冷却何时结束（精确到毫秒）
├── 关联物品 ID        — 如果是通过物品触发的，记录物品 ID
├── 冷却分类 ID        — 该法术所属的冷却分类
├── 分类冷却结束时间    — 分类冷却何时结束
└── OnHold 标志        — 是否处于等待触发状态
```

**OnHold 标志**是一种特殊的状态：冷却不会立即开始倒计时，而是等待某个事件触发后才转为正常冷却。最典型的场景是药水——在战斗中饮用后，冷却等到离开战斗后才开始。

**关联物品 ID**的存在是因为同一个法术可能由不同的物品触发（如不同种类的药水），需要为每个物品独立追踪冷却。

**时间精度**：冷却结束时间使用系统时钟的时间点存储，精度为毫秒级别。这确保了在高频法术场景下（如每秒多次施放）冷却判断的准确性。

### 2.2 充能条目

充能条目记录一次消费后的恢复周期——从开始恢复到恢复完成的起止时间：

```
充能条目
├── 恢复开始时间        — 这次恢复何时开始计时
└── 恢复结束时间        — 恢复完成，充能可用的时间点
```

充能条目的关键设计在于**恢复开始时间的对齐**：新消费的充能，其恢复开始时间不是"当前时刻"，而是队列中前一个充能条目的恢复结束时间。这确保了多个充能按顺序依次恢复，而非并行恢复。

### 2.3 四种存储类型

冷却管理器使用四种不同的存储结构来管理不同类型的时间状态：

| 存储类型 | 键 | 值 | 用途 |
|----------|------|------|------|
| 法术冷却存储 | 法术 ID | 冷却条目 | 按法术 ID 存储每个法术的冷却信息 |
| 分类冷却存储 | 分类 ID | 冷却条目指针 | 按分类 ID 索引，指针指向法术冷却存储中的条目 |
| 充能恢复队列 | 分类 ID | 充能条目队列 | 按分类存储充能恢复的双端队列 |
| 全局冷却存储 | 恢复分类 ID | 时间点 | 按恢复分类存储 GCD 结束时间 |

**分类冷却的指针设计**：分类冷却存储中的指针直接指向法术冷却存储中的条目，二者共享同一份数据。这意味着修改法术冷却条目时，分类冷却自动同步——无需额外的维护逻辑。

**充能队列的双端队列选择**：充能恢复队列使用双端队列（deque）数据结构，因为：
- 队首弹出（最早消费的充能最先恢复）
- 队尾追加（新消费的充能排到恢复队列末尾）
- 这两种操作在双端队列中都是常数时间复杂度

### 2.4 冷却标志

系统定义了一组标志位，用于控制客户端冷却 UI 的显示行为：

| 标志 | 含义 |
|------|------|
| 无标志 | 默认行为 |
| 包含 GCD | 客户端将此冷却显示为 GCD 类型 |
| 包含事件冷却 | 客户端将此冷却显示为事件触发类型 |
| 失控 UI | 客户端将此冷却显示在"失控"效果区域（如打断图标） |
| OnHold | 客户端知道此冷却处于等待触发状态 |

这些标志通过冷却同步包发送给客户端，让客户端能正确渲染冷却图标的不同视觉状态。

## 3 冷却管理流程

### 3.1 HandleCooldowns — 冷却处理入口

法术施放成功后，冷却管理器的入口方法会执行一系列检查来决定如何处理冷却。这个方法是一个多路分流的决策树：

**检查 1：法术是否标记为忽略冷却？**
某些法术（如被触发器触发的法术）由引擎标记为忽略冷却。如果命中，直接返回，不做任何冷却处理。

**检查 2：消费充能**
如果法术有充能分类，无论后续逻辑如何，都先消费一个充能。充能消费后追加到恢复队列中。

**检查 3：是否有"忽略冷却"光环？**
某些光环效果可以完全跳过冷却。如果单位身上有此类光环，直接返回。

**检查 4：药水或事件触发冷却的特殊处理？**
药水在战斗中使用后不立即冷却，而是进入 OnHold 状态——等到离开战斗后才开始冷却。类似地，某些法术标记了"冷却在特定事件发生时启动"，也进入 OnHold 状态。

**检查 5：是否是被动法术？**
被动法术不需要冷却——它们是常驻效果。

**检查 6：正常启动冷却**
如果通过了所有检查，调用冷却启动方法，进入修正器链计算最终冷却时间。

### 3.2 OnHold 事件触发冷却

OnHold 是一种延迟冷却机制，适用于两类场景：

**药水场景**：
- 玩家在战斗中饮用药水
- 冷却不会立即开始（否则玩家在战斗中会浪费药水冷却时间）
- 离开战斗后，冷却才开始倒计时
- 这让玩家可以在下次战斗中再次使用药水

**事件触发场景**：
- 某些法术标记了"冷却在事件发生时启动"
- 具体事件由游戏逻辑定义（如切换目标、受到伤害等）

**OnHold 的实现方式**：
- 冷却结束时间被设为一个极长的延迟（约一个月），实质上是"无限期"
- OnHold 状态的冷却不会被持久化到数据库——如果玩家在 OnHold 状态下登出，冷却会丢失
- OnHold 状态的冷却在决斗恢复时不会被覆盖——保持其等待触发的状态

### 3.3 StartCooldown — 冷却启动与修正器链

冷却启动是冷却管理中最核心的方法。它从获取基础冷却时间开始，经过一系列修正器的叠加，计算出最终冷却时间：

```
获取基础冷却时间
    │
    │  优先级: 物品效果数据 > 法术基础数据
    │
    ▼
[OnHold?] ──→ 是 ──→ 设为极长延迟
    │
    否
    ▼
法术修正器（SPELL_MOD_COOLDOWN）
    │  按固定值增减冷却时间
    ▼
分类冷却修正器
    │  按固定值增减分类冷却时间
    ▼
急速影响冷却
    │  急速越高，冷却越短
    ▼
急速恢复影响冷却
    │  另一种急速加成来源
    ▼
冷却恢复速率修正
    │  按乘数调整冷却速度
    ▼
按标签的恢复速率修正
    │  只对特定标签的法术生效
    ▼
固定冷却时间增减
    │  直接加减固定毫秒数
    ▼
分类冷却增减
    │  直接加减分类冷却的固定毫秒数
    ▼
[分类冷却以天为单位?] ──→ 是 ──→ 使用每日任务重置时间
    │
    否
    ▼
[修正后冷却为 0?] ──→ 是 ──→ 无冷却，结束
    │
    否
    ▼
注册到法术冷却存储和分类冷却存储
    │
    ▼
[需要通知客户端?] ──→ 是 ──→ 发送冷却同步包
```

**修正器叠加顺序的重要性**：修正器的执行顺序直接影响最终冷却时间。例如，"固定值增减"如果放在"乘数修正"之前，效果会与放在之后不同。TrinityCore 的修正器链顺序是经过精心设计的，确保了急速加成、固定增减、分类增减等效果的正确叠加。

**冷却时长来源的优先级**：
- 第一优先级：物品效果数据。如果法术是由物品触发的，物品本身可能定义了独立的冷却时间
- 第二优先级：法术基础数据。法术原型数据中定义的默认冷却时间

**每日重置分类冷却**：某些分类冷却以"天"为单位（如每日任务奖励的物品），其冷却结束时间对齐到每日重置时刻，而非固定的小时数。

### 3.4 全局冷却（GCD）

全局冷却是一种特殊的冷却——几乎所有法术都受其约束。当施放任何法术时，属于同一恢复分类的所有法术都会进入短暂的全局冷却。

**GCD 的存储方式**：
- 以"恢复分类"为键存储
- 值为 GCD 结束时间点
- GCD 和法术冷却是独立存储、独立管理的

**GCD 时钟的特殊性**：
- GCD 的检查使用系统时钟
- 法术冷却的检查使用游戏时钟
- 这意味着在服务器时间暂停的场景下，GCD 可能不受影响，而法术冷却会被冻结

**GCD 操作接口**：

| 操作 | 用途 |
|------|------|
| 检查 GCD | 判断指定法术是否处于 GCD 中 |
| 添加 GCD | 为指定恢复分类设置 GCD |
| 取消 GCD | 取消指定恢复分类的 GCD（如某些光环效果） |
| 获取剩余 GCD | 获取指定恢复分类的剩余 GCD 时间 |

### 3.5 法术系封锁

法术系封锁发生在施法被打断时（如法师的法术反制），会锁定指定法术系的所有法术：

**存储方式**：
- 使用固定大小的数组，每个元素对应一个法术系的封锁结束时间
- 法术系是一个枚举值（如火焰、冰霜、奥术、神圣、暗影等）

**封锁的执行流程**：

```
施法被打断（打断效果触发）
    │
    ▼
遍历被封锁的法术系位
    │
    ▼
收集该单位已知的所有法术
    │  包括: 玩家已学法术、宠物法术、生物法术
    │
    ▼
对每个已知法术检查:
    ├── 该法术是否受沉默预防类型影响？
    ├── 该法术是否标记为忽略封锁？
    └── 该法术是否属于被封锁的法术系？
    │
    ├── 以上任一条件满足 ──→ 跳过
    └── 都不满足 ──→ 比较封锁时间与已有冷却
           │
           ├── 封锁时间更长 ──→ 注册新冷却（覆盖旧冷却）
           └── 已有冷却更长 ──→ 保留旧冷却
    │
    ▼
发送带"失控 UI"标志的冷却包
```

**封锁时间覆盖原则**：只有当新的封锁时间长于已有冷却的剩余时间时，才会覆盖。这防止了短时间的打断效果重置已经存在的长冷却。

**法术系封锁的客户端显示**：被打断的法术冷却包带有"失控 UI"标志，客户端会在施法条上显示特殊的"被封锁"图标，而非普通的冷却旋转效果。

### 3.6 定时清理

冷却管理器在每个游戏时间 tick 中执行清理操作，移除已经过期的条目：

```
Update() — 每 tick 执行
    │
    ├── 清理分类冷却: 删除分类冷却结束时间早于当前时间的条目
    │
    ├── 清理法术冷却: 删除法术冷却结束时间早于当前时间的条目
    │   同时移除分类冷却存储中指向该条目的指针
    │
    └── 清理充能队列: 弹出队首恢复结束时间早于当前时间的充能条目
```

这种惰性清理策略避免了为每个冷却条目设置定时器带来的开销。每 tick 扫描一次，过期条目被批量清除。

## 4 充能系统

### 4.1 串行恢复模型

充能系统允许某些法术拥有多次使用次数。每次使用消耗一个充能，充能逐个恢复。

**为什么选择串行恢复而非并行恢复？**

并行恢复意味着每次消费的充能都独立计时、同时恢复。串行恢复则要求充能按消费顺序排队，前一个恢复完成后后一个才开始恢复。

```
串行恢复（TrinityCore 的选择）:

时间轴 ─────────────────────────────────────▶

初始状态 (3 次充能):
  [可用] [可用] [可用]

第 1 次施法 (t=0):
  [恢复中...] [可用] [可用]
  队列: [0s ──── 5s]

第 2 次施法 (t=1s):
  [恢复中...] [恢复中...] [可用]
  队列: [0s ── 5s][5s ──── 10s]
                  ^── 对齐到前一个的结束时间

第 3 次施法 (t=2s):
  [恢复中...] [恢复中...] [恢复中...]
  队列: [0s ── 5s][5s ── 10s][10s ── 15s]

t=5s:  [可用]     [恢复中...] [恢复中...]
t=10s: [可用]     [可用]     [恢复中...]
t=15s: [可用]     [可用]     [可用]
```

**串行恢复的设计优势**：

- **可预测的恢复节奏**：玩家知道每隔固定时间恢复一个充能，形成稳定的节奏感
- **简化 UI 展示**：客户端只需展示一个"倒计时"而非多个独立的计时器
- **平衡性更好**：串行恢复的总恢复时间更长，为数值平衡提供了更大的调整空间
- **实现简洁**：双端队列天然支持"先进先出"的恢复顺序

### 4.2 充能消费

当法术被施放时，冷却管理器会消费一个充能：

```
ConsumeCharge(充能分类 ID)
    │
    ├── 分类不存在？ ──→ 返回（无需充能）
    │
    ├── 恢复时间 ≤ 0 或最大充能 ≤ 0？ ──→ 返回（充能机制未启用）
    │
    ├── 有"忽略充能冷却"光环？ ──→ 返回
    │
    ├── 恢复队列为空？
    │       │
    │   是 ──→ 恢复开始时间 = 当前时刻
    │   否 ──→ 恢复开始时间 = 队尾条目的恢复结束时间
    │
    ▼
  创建新充能条目，追加到队列尾部
```

**对齐到队尾**：这是串行恢复的核心实现。新消费的充能，其恢复开始时间等于队列中最后一个条目的恢复结束时间，而非当前时刻。即使当前时刻已经过了前一个充能的恢复结束时间，对齐行为也不会改变。

### 4.3 充能恢复时间计算

充能恢复时间的基础值来自法术分类数据，然后经过一系列修正器计算最终值：

```
基础恢复时间 = 分类定义的充能恢复时间
    │
    ▼
+ 固定时间增减
    │  直接加减固定毫秒数
    ▼
+ 按类型掩码增减
    │  只对特定类型的法术生效
    ▼
× 乘数修正
    │  按百分比调整恢复时间
    ▼
[有急速影响？] ──→ 是 ──→ 按急速缩短恢复时间
    │
[有急速恢复影响？] ──→ 是 ──→ 按急速恢复缩短恢复时间
    │
    ▼
× 按分类 ID 的恢复速率
    │
× 按类型掩码的恢复速率
    │
    ▼
[基础恢复时间 ≤ 1 小时 且 非忽略时间速率？]
    │
    ▼
应用全局时间速率
    │
    ▼
取整（向下取整）
```

**充能修正器链与冷却修正器链的对比**：

充能修正器链与法术冷却修正器链类似，但有以下区别：
- 充能修正器链有"按类型掩码增减"和"乘数修正"，冷却修正器链没有
- 充能修正器链的"急速影响"通过独立的光环类型控制，而冷却修正器链的急速是统一应用
- 充能修正器链的最后一步有"全局时间速率"检查，只在基础恢复时间不超过 1 小时时生效

### 4.4 充能可用性检查

判断法术是否还有可用充能的逻辑：

```
HasCharge(充能分类 ID)
    │
    ├── 分类不存在？ ──→ 返回 true（无需充能即可施放）
    │
    ├── 最大充能 ≤ 0？ ──→ 返回 true（充能机制未启用）
    │
    ├── 查找恢复队列
    │
    ├── 队列为空？ ──→ 返回 true（没有已消费的充能）
    │
    └── 已消费数量 < 最大充能？ ──→ 返回 true（还有剩余）
                                  返回 false（全部已消费）
```

**设计要点**：
- 分类不存在或最大充能为零时，视为"该法术不需要充能"，始终可以施放
- 只有"已消费数量等于最大充能数"时才返回"无充能"

### 4.5 充能修改操作

除了基本的消费和恢复，系统还支持动态修改充能状态：

**修改恢复时间**：
- 对每个充能条目的开始时间和结束时间加上相同的偏移量
- 整个恢复窗口整体平移，但恢复时长不变

```
修改前: [0s ── 5s][5s ── 10s]
偏移 +3s: [3s ── 8s][8s ── 13s]
```

修改后系统会检查是否有已过期的条目需要弹出，并通知客户端更新。

**修改恢复速率**：
- 使用乘数方式调整恢复速度
- 第一个充能：基于当前时间点缩放剩余时间
- 后续充能：基于恢复时长缩放并重新链接

```
修改前 (速率减半，当前时刻 = 2s):
  [0s ── 5s][5s ── 10s]

第一个: 新结束时间 = 当前时刻 + 剩余时间 × 0.5 = 2s + 3s × 0.5 = 3.5s
后续:   新时长 = 原时长 × 0.5 = 5s × 0.5 = 2.5s
  [? ── 3.5s][3.5s ── 6.0s]
```

**恢复充能**：
- 从恢复队列尾部弹出一个条目（立即恢复一个充能）
- 通常由特定游戏事件触发（如拾取某个物品、完成某个任务）

**重置充能**：
- 清空指定分类的所有充能恢复条目
- 相当于"所有充能立即可用"

## 5 特殊场景

### 5.1 冷却暂停与恢复

冷却暂停机制允许在特定场景下冻结所有冷却计时器（如进入载具、观看过场动画等）：

**暂停操作**：
- 记录暂停开始的时间点
- 仅此而已——不修改任何冷却条目

**恢复操作**：
- 计算暂停持续时长（当前时刻 - 暂停时刻）
- 遍历所有法术冷却条目，将冷却结束时间向后推移暂停时长
- 遍历所有充能恢复条目，将恢复结束时间向后推移暂停时长
- 重置暂停标记
- 执行一次清理，移除因时间推移而过期的条目

```
暂停前 (t=0):
  法术 A 冷却: 结束于 t=10s
  法术 B 冷却: 结束于 t=5s
  充能: 恢复于 t=8s

暂停 (t=3s), 恢复 (t=7s), 暂停了 4s:
  法术 A 冷却: 结束于 t=14s (10+4)
  法术 B 冷却: 结束于 t=9s (5+4)
  充能: 恢复于 t=12s (8+4)
```

**设计细节**：恢复时只延长法术冷却结束时间和充能恢复结束时间，不单独延长分类冷却结束时间。但由于分类冷却存储中的指针指向法术冷却存储中的条目，二者自然会保持一致。

### 5.2 持久化设计

冷却状态需要在玩家登出时保存到数据库，在登录时恢复。系统采用模板化设计，支持两种所有者类型：

**模板化 Load/Save**：
- 定义一个模板函数，接受所有者类型作为模板参数
- 通过特化的辅助类提供不同的 SQL 语句和字段映射
- 玩家和宠物使用不同的数据库表

**玩家 vs 宠物的差异**：

| 维度 | 玩家 | 宠物 |
|------|------|------|
| 冷却字段数 | 5 个（含物品 ID） | 4 个（无物品 ID） |
| 充能字段数 | 3 个 | 3 个 |
| 数据库表 | 角色冷却表 | 宠物冷却表 |

**加载流程**：

```
LoadFromDB(冷却查询结果, 充能查询结果)
    │
    ├── 遍历冷却记录:
    │   ├── 解析字段
    │   ├── 验证法术数据存在性
    │   ├── 存入法术冷却存储
    │   └── 如有分类 ID，建立分类冷却指针
    │
    └── 遍历充能记录:
        ├── 解析字段
        ├── 验证分类数据存在性
        └── 追加到充能恢复队列
```

**保存流程**：

```
SaveToDB(数据库事务)
    │
    ├── 删除旧的冷却记录
    │
    ├── 遍历法术冷却存储:
    │   ├── 跳过 OnHold 状态的冷却 ← 关键设计
    │   └── 插入新记录
    │
    ├── 删除旧的充能记录
    │
    └── 遍历充能恢复队列:
        └── 插入所有充能记录
```

**OnHold 不持久化**：这是一个有意为之的设计决策。OnHold 状态的冷却由运行时事件（如战斗结束）触发，而非固定的时间倒计时。如果玩家在 OnHold 状态下登出再登入，对应的冷却会丢失——这避免了将"无限期延迟"写入数据库的语义问题。

### 5.3 决斗冷却快照

决斗系统需要保存和恢复冷却状态，确保决斗不影响玩家正常的冷却计时：

**决斗前保存**：
- 深拷贝当前所有冷却条目到独立的快照存储

**决斗后恢复**：

```
RestoreCooldownStateAfterDuel()
    │
    ├── 遍历当前冷却: 找出决斗中新增的"长冷却"
    │   (超过 10 分钟的冷却，如专业制作冷却)
    │   └── 合并到快照中（保留这些冷却）
    │
    ├── 遍历快照中的冷却:
    │   ├── 跳过 OnHold 状态的条目（不覆盖现有 OnHold）
    │   └── 写回冷却存储
    │
    └── 发送冷却同步包给客户端
        只发送 0~10 分钟范围的冷却
        （避免超长冷却导致的客户端 UI 显示异常）
```

**设计要点**：
- 决斗中触发的专业制作冷却等长冷却会被保留——这些冷却的触发与决斗无关，玩家不应因为决斗而"免费"获得专业制作的冷却重置
- OnHold 状态的冷却不会被恢复操作覆盖——决斗期间如果药水进入了 OnHold 状态，恢复后仍然保持 OnHold，等待战斗退出事件触发
- 客户端发包有范围限制（0-10 分钟）——超过 10 分钟的冷却不发送给客户端，因为客户端 UI 无法正确显示超长冷却的旋转动画

### 5.3.1 决斗冷却的设计动机

决斗是玩家间的临时对抗，不应该对长期的游戏状态产生持久影响。想象以下场景：

- 玩家 A 的终极技能有 10 分钟冷却，在决斗前刚刚使用过
- 决斗中，玩家 A 再次使用了终极技能
- 决斗结束后，如果终极技能仍然处于冷却中，玩家 A 的正常游戏体验就会受到严重影响

冷却快照机制通过"保存 → 合并 → 恢复"的流程解决了这个问题：决斗中新增的短期冷却被回滚，而长期冷却（如专业制作）则被保留。

### 5.4 IsReady — 法术可用性综合判定

判断法术是否可以施放的综合检查，按优先级依次执行三个子系统的检查：

```
IsReady(法术信息, 物品 ID)
    │
    ├── [1] 法术系封锁检查
    │   法术不忽略法术系封锁
    │   且 法术的预防类型包含"沉默"
    │   且 对应法术系被封锁？ ──→ 不可施放
    │
    ├── [2] 法术冷却检查
    │   该法术处于冷却中？ ──→ 不可施放
    │
    ├── [3] 充能检查
    │   该法术没有可用充能？ ──→ 不可施放
    │
    └── 全部通过 ──→ 可以施放
```

**优先级设计**：法术系封锁 > 法术冷却 > 充能可用性。这个顺序反映了"禁止程度"的递减——法术系封锁是最强的禁用，充能用尽是最弱的禁用（因为充能可以自动恢复）。

**法术系封锁的额外条件**：不是所有法术都会被法术系封锁影响。只有满足以下条件的法术才会被封锁：
- 法术没有标记为"忽略法术系封锁"
- 法术的预防类型包含"沉默"（即该法术属于需要"开口念咒"的类型）

这允许某些物理技能在法术系被封锁时仍然可用。

### 6.1 与法术状态机的交互

冷却系统嵌入在法术处理流程的多个关键节点：

```
法术状态机
    │
    ├── [准备阶段]
    │   └── 调用 IsReady() ──→ 检查法术系封锁 + 冷却 + 充能

    │   └── 检查 GCD
    │

    │
    ├── [完成阶段]
    │   └── 调用 HandleCooldowns() ──→ 启动冷却/消费充能
    │
    └── [载具进入/离开]
        └── 调用 PauseCooldowns() / ResumeCooldowns()
```

### 6.2 与脚本系统的交互

脚本系统可以在冷却流程中进行拦截：

- 修改冷却时间（通过法术修正器）
- 取消 GCD（通过特定光环效果）
- 阻止冷却启动（通过"忽略冷却"光环）
- 修改充能恢复速率（通过充能速率修正光环）

### 6.3 与 Aura 系统的交互

Aura 系统是冷却修正器的主要来源。多种 Aura 效果可以影响冷却和充能：

| Aura 效果 | 影响对象 | 方式 |
|-----------|---------|------|
| 忽略冷却光环 | 法术冷却 | 完全跳过冷却 |
| 急速影响冷却 | 法术冷却 | 按急速缩短 |
| 冷却恢复速率 | 法术冷却 | 按乘数调整 |
| 固定冷却增减 | 法术冷却 | 按固定值增减 |
| 分类冷却增减 | 分类冷却 | 按固定值增减 |
| 充能恢复时间增减 | 充能恢复 | 按固定值增减 |
| 充能恢复乘数 | 充能恢复 | 按百分比调整 |
| 充能恢复速率 | 充能恢复 | 按乘数调整 |
| 急速影响充能恢复 | 充能恢复 | 按急速缩短 |

## 7 复刻指南

### 7.1 关键设计决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 充能恢复模型 | 串行恢复（队列） | 可预测节奏、简化 UI、更好的平衡性 |
| 分类冷却存储 | 指针指向法术冷却 | 共享数据，无需额外同步 |
| 充能队列数据结构 | 双端队列 | 高效的队首弹出和队尾追加 |
| 修正器执行顺序 | 固定顺序链式处理 | 确保修正效果的正确叠加 |
| OnHold 状态 | 不持久化 | 由运行时事件驱动，持久化语义复杂 |
| 冷却清理策略 | 惰性清理（tick 扫描） | 避免为每个冷却设置定时器 |
| 法术系封锁存储 | 固定大小数组 | 法术系数量有限且固定，数组访问最快 |
| 时间精度 | 毫秒级时间点 | 高频法术场景需要足够精度 |
| GCD 时钟 | 系统时钟（非游戏时钟） | GCD 不受游戏时间暂停影响 |
| 决斗快照 | 深拷贝 + 合并 | 保护长冷却不被决斗覆盖 |

### 7.2 实现步骤建议

**第一步：定义核心数据结构**

创建冷却条目和充能条目结构体。冷却条目包含法术 ID、冷却结束时间、分类 ID、分类冷却结束时间和 OnHold 标志。充能条目包含恢复开始时间和恢复结束时间。

**第二步：定义存储容器**

创建四种存储：法术冷却映射（法术 ID → 冷却条目）、分类冷却映射（分类 ID → 指向法术冷却条目的指针）、充能恢复队列映射（分类 ID → 双端队列）、GCD 映射（恢复分类 ID → 时间点）。

**第三步：实现冷却启动入口**

实现从法术施放到冷却注册的完整分流逻辑：忽略冷却检查、充能消费、光环忽略检查、药水/事件触发检查、被动法术检查、正常冷却启动。

**第四步：实现修正器链**

按固定顺序实现冷却时间的修正器链。每个修正器查询对应的光环效果并计算修正值。注意叠加顺序——先增减后乘数、先分类后全局。

**第五步：实现充能系统**

实现充能消费（对齐到队尾）、恢复队列清理（队首弹出）、可用性检查、恢复时间计算（含修正器链）、速率修改。

**第六步：实现 GCD 和法术系封锁**

GCD 按恢复分类管理，独立于法术冷却。法术系封锁使用固定数组，打断时遍历已知法术注册封锁冷却。

**第七步：实现 IsReady 综合判定**

按法术系封锁 > 法术冷却 > 充能可用性的优先级检查。注意法术系封锁需要额外检查预防类型。

**第八步：实现持久化**

实现模板化的数据库 Load/Save。OnHold 状态的冷却不持久化。支持玩家和宠物两种所有者类型。

**第九步：实现特殊场景**

实现冷却暂停/恢复（整体推移时间点）、决斗快照（深拷贝 + 合并恢复）。

### 7.3 常见陷阱

**陷阱一：分类冷却与法术冷却不一致**

如果分类冷却和法术冷却存储独立的数据，修改一个时必须同步修改另一个。TrinityCore 通过指针共享解决了这个问题。如果你选择独立存储，必须确保所有修改路径都包含同步逻辑。

**陷阱二：修正器叠加顺序错误**

修正器的叠加顺序直接影响最终结果。"先乘后加"和"先加后乘"会产生不同的数值。建议严格遵循"先固定值增减，后乘数修正"的原则，并在文档中明确记录叠加顺序。

**陷阱三：充能恢复时间对齐边界**

当充能消费速度很快（如每秒多次），恢复开始时间的对齐可能导致队列快速积压。建议在实现时添加队列长度上限检查，或考虑合并相近的恢复条目。

**陷阱四：暂停恢复后的时钟差异**

如果暂停恢复期间游戏时间被调整（如服务器时间校准），恢复后的时间推移可能与实际暂停时长不一致。建议在暂停时记录游戏时间，恢复时计算游戏时间差（而非系统时间差）。

**陷阱五：GCD 时钟与法术冷却时钟不一致**

GCD 使用系统时钟而法术冷却使用游戏时钟，在时间暂停场景下二者的行为不同。如果你的项目不需要时间暂停功能，建议统一使用同一个时钟源。

## 8 总结

TrinityCore 的冷却与充能管理系统是一个精密的时间状态管理器，它统一了法术冷却、分类冷却、充能恢复、GCD 和法术系封锁五个不同的时间维度。

这个系统的设计体现了几个重要的工程原则：

**统一入口，分类处理**：所有冷却操作都通过一个入口方法进入，内部根据法术属性分流到不同的处理路径。这使得新增加的特殊场景（如 OnHold）只需要在分流逻辑中添加一个分支。

**修正器链的幂等性**：每个修正器只负责一种维度的修正，通过固定的执行顺序保证叠加的正确性。新增修正器只需在链中的合适位置插入。

**串行恢复的优雅**：充能系统使用双端队列和对齐策略，用极简的数据结构实现了有序恢复。没有定时器、没有回调、没有事件——只需在每 tick 弹出过期的队首条目。

**指针共享的一致性**：分类冷却存储通过指针指向法术冷却存储中的条目，消除了数据同步的复杂性。一个修改自动传播到两个视图。

对于希望在自己的项目中复刻类似冷却系统的工程师，建议从充能恢复队列开始实现——它是整个系统中最有趣也最有工程价值的部分。双端队列加队尾对齐的策略，可以用于任何需要"有序恢复"的游戏系统。
