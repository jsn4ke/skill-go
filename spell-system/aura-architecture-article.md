# MMO Aura 系统架构：完整设计

> 一份面向游戏服务端工程师的 Aura 系统架构设计参考。本文基于对 TrinityCore（World of Warcraft 私有服务器）Aura 系统的深度分析，以纯概念方式描述其架构，使读者能够在自己的项目中复刻类似系统。本文是施法状态机、效果管线、目标选择文章的姊妹篇。

---

## 1 引言

### 1.1 什么是 Aura

在 MMO 中，"法术"（Spell）是一次性的——施法、命中、效果执行、结束。但很多效果需要**持续存在**：一个持续 30 秒的护盾、一个每 3 秒造成伤害的毒药、一个增加 20% 攻击力的增益。

**Aura** 就是这种"持续性效果"的统一抽象。当法术效果类型为"施加 Aura"时，法术命中后不在目标身上一次性执行，而是创建一个 Aura 对象，持续对目标施加影响。

```
┌─────────────────────────────────────────────────────────────┐
│  Spell（瞬态）                                               │
│  生命周期：从施法开始到效果执行完毕                             │
│  典型时长：几百毫秒                                            │
└──────────────────┬──────────────────────────────────────────┘
                   │ 法术命中 → 效果类型 = 施加 Aura
                   ▼
┌─────────────────────────────────────────────────────────────┐
│  Aura（持久态）                                               │
│  生命周期：从创建到过期/移除                                    │
│  典型时长：几秒到几分钟                                        │
│                                                              │
│  持续执行：                                                   │
│  ├── 周期性触发（Tick）—— 每 N 秒造成伤害/治疗                 │
│  ├── Proc 触发 —— 满足条件时自动触发效果                       │
│  ├── 持续时间倒计时 —— 到期后自动移除                          │
│  └── 层数/充能管理 —— 支持可叠加和可消耗的效果                  │
└─────────────────────────────────────────────────────────────┘
```

**关键区分**：Spell 是"瞬态过程"，Aura 是"持久化状态"。一个 Spell 可以产生多个 Aura（如多目标 AOE），一个 Aura 可以包含多个效果（如同时减速 + 减攻击速度）。

### 1.2 Aura 与效果管线的关系

Aura 的创建是效果管线的产物。当效果管线在 HitTarget 阶段遇到"施加 Aura"类型的效果时，它会调用 Aura 系统的创建入口：

```
效果管线 (HitTarget 阶段)
      │
      ├── 效果类型 = 造成伤害 ──→ 直接扣血
      ├── 效果类型 = 治疗     ──→ 直接加血
      ├── 效果类型 = 施加 Aura ──→ 创建 Aura 对象
      └── 其他效果类型        ──→ 对应处理器
```

Aura 创建后独立于法术运行。即使原始法术已经结束、施法者已经死亡，Aura 仍然会继续执行直到过期。

### 1.3 本文目标读者

- 希望在自己的项目中实现类似 Buff/Debuff 系统的服务端工程师
- 需要理解"持久化效果"架构的架构师
- 正在评估三层架构 vs 两层架构等设计方案的决策者

## 2 三层架构

### 2.1 为什么需要三层

初看之下，一个 Aura 只需要一个对象就够了——存储效果类型、数值、持续时间。但实际需求很快暴露了两层不足的问题：

- **一个法术多种效果**：一个法术可能同时减速 30% 和降低攻击速度 20%。这是两个独立的效果，各有各的数值和持续时间
- **一个效果多个目标**：一个 AOE 法术可能在多个目标身上同时生效。每个目标需要独立的状态追踪（如不同的效果掩码）

因此系统被设计为**三层架构**：

```
┌──────────────────────────────────────────────────────┐
│ Aura（效果容器）                                      │
│ 身份：法术 ID + 施法者 ID + 施法物品 ID               │
│ 状态：持续时间、层数、充能数                          │
│ 持有者：一个 Unit 或 DynamicObject                    │
│                                                      │
│   ┌──────────────────┐  ┌──────────────────┐         │
│   │ AuraEffect #0    │  │ AuraEffect #1    │         │
│   │ 类型：减速       │  │ 类型：减攻速     │         │
│   │ 数值：30%        │  │ 数值：20%        │         │
│   │ 周期：无         │  │ 周期：无         │         │
│   └──────────────────┘  └──────────────────┘         │
│                                                      │
│   ┌──────────────────┐  ┌──────────────────┐         │
│   │ AuraApplication  │  │ AuraApplication  │         │
│   │ → 目标 A         │  │ → 目标 B         │         │
│   │ 槽位：12         │  │ 槽位：15         │         │
│   │ 效果掩码：11     │  │ 效果掩码：10     │         │
│   └──────────────────┘  └──────────────────┘         │
└──────────────────────────────────────────────────────┘
```

**设计决策：为什么三层而不是两层？**

| 对比 | 两层（Aura + Effect） | 三层（Aura + Effect + Application） |
|------|----------------------|-----------------------------------|
| 多效果支持 | 需要在 Aura 中存储多个效果的数组 | AuraEffect 自然支持 |
| Area Aura 多目标 | 需要在 Aura 中维护目标列表和每个目标的状态 | AuraApplication 自然支持 |
| 状态追踪粒度 | 粗粒度（Aura 级别） | 细粒度（Application 级别） |
| 复杂度 | 低 | 中等 |

两层架构的局限在于：当 Area Aura 的不同目标需要不同的效果掩码时（如 Boss 免疫了减速但没免疫减攻速），两层架构无法表达这种"同一 Aura 在不同目标上效果不同"的需求。

### 2.2 Aura 层（效果容器）

Aura 层是整个系统的**核心中枢**，管理效果、应用实例、持续时间和 Proc。

**职责清单**：

| 职责 | 说明 |
|------|------|
| 身份管理 | 存储法术 ID、施法者 ID、施法物品 ID，唯一标识一个 Aura |
| 持续时间 | 当前剩余时间、最大持续时间、刷新计时 |
| 层数管理 | 当前层数、最大可堆叠层数 |
| 充能管理 | Proc 充能次数、充能消耗与补充 |
| 效果管理 | 持有 AuraEffect 数组（最多等于法术效果槽数） |
| 目标管理 | 持有 AuraApplication 映射（按目标 GUID 索引） |
| 堆叠判定 | 决定新 Aura 能否与已有 Aura 共存 |
| Proc 系统 | 监听事件、判定触发概率、执行触发效果 |
| 客户端同步 | 标记需要更新的目标，通知客户端刷新显示 |

**两个子类**：UnitAura（持有者为游戏单位）和 DynObjAura（持有者为动态对象，如图腾、冰环）。DynObjAura 不支持递减效果（Diminishing Returns）和心跳（Heartbeat）。

### 2.3 AuraEffect 层（效果实例）

AuraEffect 代表 Aura 中的**单个效果**。一个法术最多有 N 个效果槽位，每个槽位对应一个 AuraEffect。

**职责清单**：

| 职责 | 说明 |
|------|------|
| 效果类型 | 存储效果类型（伤害、治疗、属性修改、控制等） |
| 效果数值 | 基础数值和当前实际数值（含系数计算） |
| 周期管理 | 周期间隔、周期计时器、已完成 Tick 数 |
| 效果应用 | HandleEffect —— 注册/注销效果到目标的效果系统 |
| 效果分发 | 根据 AuraType 调用对应的效果处理器 |
| 数值变更 | ChangeAmount —— 更新数值并重新应用所有目标 |

**效果处理流程**（HandleEffect）：

```
HandleEffect(应用/移除)
      │
      ├── 注册/注销效果到目标的效果系统
      │     （如属性修改效果需要更新目标的属性值）
      │
      ├── 应用/移除数值修正（SpellMod）
      │
      ├── 脚本 BeforeApply/BeforeRemove 钩子
      │     └── 脚本可以阻止默认效果
      │
      ├── 默认效果处理器
      │     └── 根据 AuraType 分发到对应处理函数
      │
      └── 脚本 AfterApply/AfterRemove 钩子
```

### 2.4 AuraApplication 层（应用实例）

AuraApplication 表示 Aura 在某个**具体目标上的应用**。对于单体 Aura，只有一个 AuraApplication（施放者自己或指定目标）。对于 Area Aura，有多个 AuraApplication（区域内的每个目标各一个）。

**职责清单**：

| 职责 | 说明 |
|------|------|
| 目标引用 | 指向应用目标的指针 |
| 效果掩码 | 标记哪些效果对这个目标生效（不同目标可能有不同的掩码） |
| 槽位管理 | Aura 在目标上的显示槽位（客户端用槽位显示 Buff 图标） |
| 正负标志 | 标记是增益（Buff）还是减益（Debuff） |
| 移除原因 | 记录为什么被移除（过期、驱散、死亡等） |

**效果掩码**是 AuraApplication 最重要的字段。它解决了"同一 Aura 在不同目标上效果不同"的需求：

```
Aura（冰霜新星 AOE）
  ├── 效果 #0：造成冰霜伤害
  └── 效果 #1：降低移动速度

目标 A（普通小怪）：
  AuraApplication → 效果掩码 = 11（两个效果都生效）
  → 受到冰霜伤害 + 降低移动速度

目标 B（Boss，免疫减速）：
  AuraApplication → 效果掩码 = 01（只有效果 #0 生效）
  → 只受到冰霜伤害，不减速
```

## 3 Aura 生命周期

### 3.1 创建入口

Aura 的创建从效果管线的 HitTarget 阶段开始。当效果类型为"施加 Aura"时，管线调用 Aura 系统的统一入口：**TryRefreshStackOrCreate**（尝试刷新/堆叠或创建新 Aura）。

这个入口的处理逻辑是：

```
TryRefreshStackOrCreate
      │
      ├── 过滤效果掩码（某些效果可能对 Aura 持有者不适用）
      │
      ├── 搜索已有 Aura（同法术 ID + 同施法者 + 同物品）
      │     │
      │     ├── [找到] → 尝试堆叠/刷新 → 返回已存在的 Aura
      │     └── [未找到] → 继续创建新 Aura
      │
      └── 创建新 Aura
            │
            ├── 根据持有者类型创建 UnitAura 或 DynObjAura
            ├── 初始化所有 AuraEffect
            ├── 为持有者创建 AuraApplication
            ├── 加入持有者的 Aura 列表
            └── 移除互斥的已有 Aura
```

### 3.2 移除原因

Aura 可以因多种原因被移除，移除原因会影响脚本钩子的行为：

| 移除原因 | 触发场景 | 脚本可感知 |
|---------|---------|-----------|
| 过期 | 持续时间倒计时到 0 | 是 |
| 驱散 | 被敌方净化/驱散法术移除 | 是 |
| 死亡 | 目标死亡 | 是 |
| 取消 | 玩家右键点击 Buff 取消 | 是 |
| 中断 | 被移动、施法等行为打断 | 是 |
| 默认 | 脚本主动移除、堆叠替换 | 是 |
| 充能耗尽 | Proc 充能用完 | 是 |

### 3.3 持续时间管理

Aura 的持续时间管理支持多种操作：

| 操作 | 行为 | 典型场景 |
|------|------|---------|
| 设置 | 直接设置当前剩余时间 | 初始化 |
| 刷新 | 重新计算并设置为最大持续时间 | 重新施加同类 Aura |
| 刷新计时器 | 同时刷新持续时间和周期 Tick 计时器 | 堆叠刷新 |

**特殊规则**：
- 被动法术没有持续时间条目时，持续时间为永久（-1）
- 充能消耗完毕后通过减少层数触发移除（层数归零 → 移除）

### 3.4 层数管理

层（Stack）是 Aura 的可叠加计数。每次重新施加同类 Aura 时，层数可能增加。

```
ModStackAmount(增量)
      │
      ├── 计算新层数 = 当前层数 + 增量
      │
      ├── 计算最大可堆叠层数
      │     └── 超过上限则截断
      │
      ├── 层数 ≤ 0？
      │     └── 是 → 移除 Aura
      │
      ├── 新层数 >= 旧层数 且 可堆叠？
      │     └── 是 → 刷新持续时间和周期计时器
      │
      └── 通知客户端更新
```

**关键区分：层数（Stack）与充能（Charge）**

| 特性 | 层数 | 充能 |
|------|------|------|
| 用途 | 表示效果强度（如 3 层中毒） | 表示触发次数（如 5 次充能） |
| 增加方式 | 重新施加同类 Aura | 被动积累或特殊机制 |
| 减少方式 | 被驱散/移除时全部消失 | 每次 Proc 触发消耗一次 |
| 归零行为 | Aura 被移除 | Aura 被移除 |
| 典型例子 | 减速效果叠加至 3 层 | "下次攻击额外造成伤害"（5 次充能） |

### 3.5 Area Aura 目标更新

Area Aura（区域光环）需要周期性更新目标列表，因为目标可能进入或离开区域。

**更新周期**：每 500ms 一次。这个值是性能与实时性的平衡——更短的间隔会增加服务器负载（每次都需要执行区域搜索），更长的间隔会导致目标进出区域时有明显延迟。

**特殊处理规则**：
- 新目标需要通过免疫检查、堆叠检查和排他性检查才能被加入
- 飞行中的目标不会被 DynObjAura 命中（防止对空中的玩家施加地面效果）
- 旧目标离开区域后会被移除，但移除时会正确反向执行效果（恢复属性、移除控制等）

**更新流程**：

```
UpdateTargetMap（每 500ms）
      │
      ├── 填充目标列表
      │     ├── UnitAura：基于效果半径搜索周围单位
      │     └── DynObjAura：基于动态对象位置搜索
      │
      ├── 对比新旧目标列表
      │     ├── 旧目标不在新列表中 → 标记移除
      │     ├── 新目标不在旧列表中 → 标记添加
      │     └── 目标免疫 → 标记移除
      │
      ├── 处理移除目标
      │     └── 反向执行效果、移除 Application
      │
      └── 处理新增目标
            ├── 免疫检查
            ├── 堆叠检查
            └── 创建 Application、应用效果
```

## 4 堆叠与刷新决策

### 4.1 核心问题

当一个新的 Aura 试图施加到一个已有同类 Aura 的目标上时，系统需要回答：**是共存还是替换？** 这个决策由 CanStackWith 判定。

### 4.2 堆叠决策树

```
新 Aura 遇到已有 Aura → CanStackWith 判定
      │
      ├── 是同一个 Aura？ → 允许共存（自身刷新）
      │
      ├── 两者都是 DynObjAura 且同施放者？ → 不允许（替换）
      │
      ├── 都是被动法术、同施放者、同等级？ → 不允许（替换）
      │
      ├── 已有 Aura 的触发法术就是新 Aura 的法术 ID？ → 允许（防触发源被移除）
      │
      ├── 新 Aura 的触发法术就是已有 Aura 的法术 ID？ → 允许（同上）
      │
      ├── 有专有互斥规则？ → 不允许（替换）
      │
      ├── 技能组堆叠规则检查
      │     ├── EXCLUSIVE → 不允许
      │     ├── EXCLUSIVE_HIGHEST → 已有 Aura 等级 >= 新的则不允许
      │     ├── EXCLUSIVE_FROM_SAME_CASTER → 同施放者则不允许
      │     └── DEFAULT → 继续
      │
      ├── 不同法术家族（SpellFamilyName）？ → 允许（不同家族共存）
      │
      ├── 不同施放者？
      │     ├── 都是 DOT/HOT 类周期效果？ → 允许（不同施放者的 DoT 可共存）
      │     ├── 都是引导法术？ → 允许
      │     └── 其他 → 继续检查
      │
      ├── 载具控制效果且无空座位？ → 不允许
      │
      ├── 同一法术等级链且同施放者？
      │     ├── 多槽位且非区域效果？ → 允许
      │     ├── 不同附魔物品？ → 允许
      │     └── 其他 → 不允许
      │
      └── 默认 → 允许共存
```

**为什么堆叠规则如此复杂？** 因为游戏设计中有大量特殊规则需要尊重：
- 不同施放者的 DoT 可以共存（牧师 A 和牧师 B 的恢复术可以同时作用）
- 同一施放者的同类法术默认不共存（不能给目标上两个一样的减速）
- 但某些法术允许叠加（如中毒效果可以叠加至多 5 层）
- 触发法术需要特殊处理（防止触发源被触发的效果移除）
- 技能组可以有互斥关系（同时只能有一个光环）

### 4.3 刷新 vs 创建 vs 替换

决策结果有三种可能：

| 结果 | 行为 | 客户端表现 |
|------|------|-----------|
| **共存** | 新旧 Aura 同时存在 | 目标身上多一个 Buff 图标 |
| **刷新/堆叠** | 新 Aura 的参数合并到已有 Aura | Buff 图标的层数增加 |
| **替换** | 移除旧 Aura，创建新 Aura | Buff 图标闪烁/替换 |

**刷新/堆叠的时机**：当 CanStackWith 返回 false 但找到同 ID 的已有 Aura 时，不是简单地丢弃新 Aura，而是尝试将新 Aura 的参数"合并"到已有 Aura 上（增加层数、刷新持续时间）。

## 5 效果类型分类

### 5.1 效果类型概览

Aura 支持数百种效果类型（AuraType），可以归类为以下主要类别：

| 类别 | 核心行为 | 典型效果 |
|------|---------|---------|
| **属性修改** | 修改目标的属性值（攻击力、防御力、速度等） | 增加力量、降低移动速度 |
| **周期性伤害** | 每 N 秒造成一次伤害 | 中毒 DoT、灼烧 |
| **周期性治疗** | 每 N 秒恢复一次生命 | 恢复 HoT、回春术 |
| **周期性触发** | 每 N 秒触发一次法术 | 周期性施放效果 |
| **控制效果** | 限制目标行为 | 眩晕、沉默、定身、减速 |
| **伤害护盾** | 吸收一定量的伤害 | 真言术护盾、法力护盾 |
| **伤害分摊** | 将伤害分散到多个目标 | 灵魂链接 |
| **触发效果** | 在特定条件下触发法术 | 攻击有概率触发额外效果 |
| **变身/形态** | 改变目标的外形或模型 | 变形术 |
| **免疫/抗性** | 提供特定类型的免疫或抗性 | 无敌、魔法免疫 |
| **资源恢复** | 周期性恢复法力/能量 | 法力回复 |
| **控制解除** | 免疫或解除特定控制效果 | 自由行动、无敌 |

### 5.1.1 属性修改类

属性修改是最常见的 Aura 效果类型之一。它直接修改目标的基础属性值。

**核心机制**：效果应用时将修改量注册到目标的"属性修正系统"中，效果移除时注销。目标的属性值是"基础值 + 所有 Aura 修正的累加"。

**关键设计点**：
- 修改可以是**固定值**（+100 攻击力）或**百分比**（+20% 攻击力）
- 百分比修改基于目标的基础属性值计算
- 同类属性修改可以叠加（多个 Aura 同时增加攻击力）
- 效果数值可以在 Aura 生命周期内动态调整（如层数增加时数值变大）

### 5.1.2 控制效果类

控制效果（眩晕、沉默、减速等）是 PvP 游戏的核心机制，其设计需要考虑公平性和可反制性。

**控制效果分类**：

| 控制类型 | 行为 | 可否驱散 | 递减 |
|---------|------|---------|------|
| 眩晕 | 完全失去控制，无法行动 | 是 | 是 |
| 沉默 | 无法施放法术 | 是 | 是 |
| 定身 | 无法移动，但可以施法 | 是 | 是 |
| 减速 | 移动速度降低 | 是 | 通常无 |
| 变形 | 变为特定形态，失去大部分能力 | 特殊 | 通常无 |
| 魅惑 | 控制目标行为 | 是 | 否 |

**递减效果（Diminishing Returns）** 是控制效果的重要平衡机制：重复施加同类控制时持续时间递减（100% → 50% → 25% → 免疫），防止"无限控制"。递减信息与目标关联而非与 Aura 关联——即目标身上所有同类控制的递减状态是共享的。

### 5.1.3 伤害护盾与分摊类

**护盾**：在目标身上创建一个"伤害吸收池"。受到伤害时，先消耗护盾值，剩余部分才扣除生命值。护盾可以有多种来源（法术、装备、技能）。

**分摊**：将受到的伤害分散到多个目标上。典型场景是"灵魂链接"——团队成员之间分摊伤害。

护盾和分摊都通过脚本 Hook 提供扩展点，允许自定义吸收逻辑和分摊比例。

### 5.2 效果分发机制

与效果管线的函数指针分派表类似，Aura 效果也使用分派表：每个 AuraType 映射到一个效果处理函数。效果应用时根据 AuraType 直接索引分派表，调用对应的处理函数。

**新增效果类型的流程**：
1. 在 AuraType 枚举中分配新 ID
2. 实现效果处理函数（根据应用模式选择性执行逻辑）
3. 在分派表中注册函数指针
4. 配置数据中为新类型定义默认参数

### 5.3 周期性效果（Tick）

周期性效果是 Aura 最常见的类型——DoT、HoT、周期性触发等。它们共享相同的"Tick"机制。

**Tick 流程**：

```
AuraEffect::Update（每帧调用）
      │
      ├── 周期计时器倒计时
      │
      ├── 计时器归零？
      │     └── 否 → 继续等待
      │
      └── 是 → 执行 Tick
            │
            ├── 检查施法者是否仍然有效
            │     └── 施法者死亡/离线 → 跳过
            │
            ├── 计算本次 Tick 的效果值
            │     ├── 基础数值
            │     ├── 脚本 CalcDamageAndHealing 钩子
            │     └── 暴击判定（DoT/HoT 可以暴击）
            │
            └── 执行效果
                  ├── DoT → 对目标造成伤害
                  ├── HoT → 对目标恢复生命
                  └── 触发 → 创建并执行新法术
```

**周期计时器的重置**：每次 Tick 执行后，计时器重置为周期间隔。如果 Aura 被堆叠/刷新，计时器也可以被重置（取决于配置）。

## 6 Proc 触发管线

### 6.1 什么是 Proc

Proc（程序化触发）是 Aura 的被动触发机制。当满足特定条件（如攻击命中、受到伤害、施放法术）时，带有 Proc 效果的 Aura 会自动触发一次效果。

典型 Proc 例子：
- "攻击有 15% 概率造成额外火焰伤害"
- "受到近战攻击时有 50% 概率减速攻击者"
- "暴击时恢复 2% 最大法力值"

### 6.2 Proc 管线总览

Proc 管线是一个多阶段的筛选和判定流程，确保只有合法的 Proc 才会触发：

```
事件触发（攻击/施法/受到伤害...）
      │
      ▼
┌─────────────────────────────────┐
│ 第一阶段：效果掩码筛选            │
│                                 │
│ 遍历目标身上所有 AuraApplication │
│ 对每个 Aura 执行：               │
│                                 │
│ ① 有 Proc 配置？                │
│    └── 无 → 跳过                │
│                                 │
│ ② 触发法术就是自身？            │
│    └── 是 → 跳过（防自触发循环）│
│                                 │
│ ③ 有抑制属性？                  │
│    └── 是 → 跳过                │
│                                 │
│ ④ 隐身法术且不破隐？            │
│    └── 是 → 跳过                │
│                                 │
│ ⑤ 无充能可消耗？                │
│    └── 是 → 跳过                │
│                                 │
│ ⑥ Proc 冷却中？                 │
│    └── 是 → 跳过                │
│                                 │
│ ⑦ DBC 数据检查                  │
│    └── 类型掩码/法术家族不匹配   │
│        → 跳过                    │
│                                 │
│ ⑧ 条件系统检查                  │
│    └── 不满足 → 跳过            │
│                                 │
│ ⑨ 脚本 CheckProc 钩子          │
│    └── 被阻止 → 跳过            │
│                                 │
│ ⑩ 逐效果检查                    │
│    └── 效果被禁用 → 排除该效果   │
│                                 │
│ 最终产出：procEffectMask         │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ 第二阶段：概率判定                │
│                                 │
│ 基础概率 = 配置的触发概率        │
│                                 │
│ 是否有 PPM 设置？                │
│ └── 是 → 用 PPM 公式替换概率    │
│                                 │
│ 是否有等级衰减？                 │
│ └── 是 → 概率 × 衰减系数        │
│                                 │
│ 天赋/装备是否修改概率？          │
│ └── 是 → 应用修改                │
│                                 │
│ 投掷随机数                       │
│ └── 失败 → 丢弃                  │
│                                 │
│ 最终产出：是否触发               │
└──────────────┬──────────────────┘
               │ [成功]
               ▼
┌─────────────────────────────────┐
│ 第三阶段：触发执行                │
│                                 │
│ ① 脚本 PrepareProc 钩子         │
│    └── 被阻止 → 中止            │
│                                 │
│ ② 预扣充能                      │
│    └── 消耗一次充能              │
│                                 │
│ ③ 设置 Proc 冷却                │
│    └── 防止短期内重复触发        │
│                                 │
│ ④ 脚本 Proc 钩子                │
│    └── 执行自定义逻辑            │
│                                 │
│ ⑤ 逐效果执行 Proc Handler       │
│    └── 触发法术、触发伤害等      │
│                                 │
│ ⑥ 脚本 AfterProc 钩子           │
│                                 │
│ ⑦ 最终消耗充能或层数            │
│    └── 充能耗尽 → 移除 Aura     │
└─────────────────────────────────┘
```

### 6.3 PPM 机制

PPM（Procs Per Minute，每分钟触发次数）是一种基于时间的 Proc 概率调节机制，使触发频率趋于稳定而非完全随机。

**核心公式**：

```
PPM 概率 = max(1.0, 1.0 + (距上次成功时间 / 平均间隔 - 1.5) × 3.0)
         × PPM值 × 距上次尝试时间 / 60.0
```

**PPM 的关键特性**：

- **运气保护**：距离上次成功触发越久，概率越高。这防止了"运气极差长时间不触发"的情况
- **时间上限**：距上次尝试最长按 10 秒计算，距上次成功最长按 1000 秒计算
- **最低概率**：基础概率最低为 1.0，确保不会概率为 0
- **自适应**：PPM 考虑了武器速度——快武器触发次数多但每次概率低，慢武器触发次数少但每次概率高

**传统概率 vs PPM**：

| 特性 | 传统概率 | PPM |
|------|---------|-----|
| 触发频率 | 波动大（可能连续触发或长时间不触发） | 趋于稳定 |
| 武器速度影响 | 无 | 有（自动调整） |
| 运气保护 | 无 | 有 |
| 适用场景 | 固定概率触发（如"100% 暴击时触发"） | 基于武器速度的触发（如"剑专精"） |

## 7 充能机制

### 7.1 充能与层数的协作

充能（Charge）和层数（Stack）是两个独立但协作的计数系统：

**充能消耗的两种模式**：

| 模式 | 行为 | 典型场景 |
|------|------|---------|
| 消耗充能 | 减少充能次数，充能耗尽后移除 Aura | "5 次充能，每次攻击消耗一次" |
| 消耗层数 | 减少层数，层数归零后移除 Aura | "5 层效果，每次触发消耗一层" |

模式由法术属性决定：如果设置了"用层数代替充能"属性，则消耗层数；否则消耗充能。

### 7.2 充能的完整生命周期

```
充能创建
      │
      ├── 计算最大充能数（由法术配置决定）
      │
      └── 初始化为最大值
            │
            ▼
      Proc 触发准备
      │
      ├── 预扣充能（非层数代替模式）
      │     └── 充能数 -1（提前锁定）
      │
      └── 执行 Proc 效果
            │
            ▼
      Proc 消耗结算
      │
      ├── 层数代替模式？
      │     ├── 是 → ModStackAmount(-1)（消耗层数）
      │     └── 否 → 充能已在准备阶段预扣
      │
      └── 充能数 == 0？
            └── 是 → 移除 Aura
```

**预扣设计**：充能在 Proc 准备阶段就预先扣除，而不是在触发完成后。这确保了如果 Proc 被脚本阻止，充能不会被白白消耗。如果准备被阻止，预扣的充能会返还。

### 7.3 延迟充能

某些场景需要"延迟增减充能"——不在当前帧立即执行，而是通过事件系统在指定延迟后执行。这用于处理一些需要时序保证的特殊机制。

## 8 脚本扩展

### 8.1 脚本 Hook 概览

Aura 系统通过 AuraScript 提供丰富的 Hook 点，覆盖 Aura 生命周期的所有关键节点。

**主要 Hook 分类**：

| 类别 | Hook | 时机 |
|------|------|------|
| 生命周期 | OnApply / OnRemove | Aura 首次应用/最终移除 |
| 效果应用 | OnEffectApply / OnEffectRemove | 每个效果单独应用/移除 |
| 周期 Tick | OnPeriodicTick | 每个 Tick 执行前 |
| 数值计算 | CalcAmount / CalcPeriodic / CalcDamageAndHealing | 效果数值计算时 |
| Proc 系统 | CheckProc / PrepareProc / OnProc / AfterProc | Proc 判定和触发的各个阶段 |
| 区域目标 | CheckAreaTarget | Area Aura 目标筛选时 |
| 吸收/分摊 | OnAbsorb / OnSplit | 伤害吸收和分摊时 |
| 战斗事件 | OnEnterLeaveCombat / OnHeartbeat | 进出战斗、心跳 |

### 8.2 SpellScript 与 AuraScript 的衔接

SpellScript 和 AuraScript 是两套独立但协作的脚本体系：

| 脚本体系 | 管辖范围 | 生命周期 |
|---------|---------|---------|
| SpellScript | 法术施放过程（准备、命中、效果执行） | 从 prepare 到 finish |
| AuraScript | Aura 持久化效果（应用、Tick、Proc、移除） | 从创建到过期 |

**时序衔接**：
- 法术命中时，SpellScript 的"命中"钩子先触发
- 然后效果管线创建 Aura
- 最后 AuraScript 的"效果应用"钩子触发

这意味着"命中"阶段的事件由 SpellScript 处理，而"应用后"的事件由 AuraScript 处理。两者不会产生冲突。

**Proc 产生的法术**：当 Aura 的 Proc 效果触发新法术时，新法术有自己独立的 SpellScript。这意味着一个 Aura 可以通过 Proc 不断触发新的法术实例，每个实例独立运行自己的脚本逻辑。

### 8.3 阻止默认行为

多个关键 Hook 支持脚本"阻止"默认行为：

- **CheckProc** → 返回 false 阻止整个 Proc 触发
- **PrepareProc** → 返回 false 阻止触发准备
- **OnProc** → 返回 true 阻止默认 Proc 处理
- **OnEffectApply** → 返回 true 阻止默认效果应用
- **OnEffectRemove** → 返回 true 阻止默认效果移除

这种"阻止"机制使得脚本可以完全替代引擎的默认行为，实现高度定制化的 Aura 效果。

## 9 设计决策回顾

### 9.1 关键决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 架构层数 | 三层（Aura/Effect/Application） | 支持多效果 + 多目标的不同效果掩码 |
| 效果分发 | 函数指针分派表 | O(1) 查找，与效果管线一致 |
| 堆叠判定 | 复杂决策树 | 游戏设计需求复杂，需要精细控制 |
| Proc 概率 | 传统概率 + PPM 双模式 | 固定概率场景用传统，武器速度相关用 PPM |
| 充能预扣 | 准备阶段预扣、被阻止则返还 | 防止被阻止时白白消耗 |
| Area Aura 更新 | 500ms 周期 | 性能与实时性的平衡 |

### 9.2 复刻建议

如果你在自己的项目中实现类似系统，建议的构建顺序：

**第一阶段：最小可用 Aura**
1. 定义 Aura 类（法术 ID、持续时间、施法者 ID）
2. 实现创建/移除流程（TryRefreshStackOrCreate 简化版）
3. 实现 2-3 种基础效果类型（属性修改、周期伤害）
4. 实现每帧更新（持续时间倒计时、Tick）

**第二阶段：多效果与多目标**
5. 拆分为三层架构（Aura/AuraEffect/AuraApplication）
6. 实现效果分派表
7. 实现 Area Aura 目标列表更新

**第三阶段：高级功能**
8. 实现堆叠决策树
9. 实现 Proc 管线（事件监听、概率判定、效果触发）
10. 实现 PPM 机制
11. 实现充能机制
12. 添加脚本 Hook 系统

**一个关键的早期决策**：你是否需要 Area Aura？如果所有效果都是单体施加的（不需要 AOE 光环），可以跳过 AuraApplication 层和 Area Aura 目标更新，大幅简化系统。两层架构（Aura + AuraEffect）在单体场景下完全够用。

**另一个关键决策**：你的游戏是否需要复杂的堆叠规则？如果所有 Buff 都是简单的"同类替换"（新 Buff 覆盖旧 Buff），堆叠逻辑可以极简化为一个"查找同名 → 移除旧 → 创建新"的流程。复杂堆叠规则（技能组互斥、不同施放者的 DoT 共存等）是 PvP 游戏的刚需，但在 PvE 单机或小规模多人游戏中可能过度设计。

**性能提示**：每个单位的 Aura 列表是高频访问的热点数据。确保使用高效的容器（如哈希表，按 AuraKey 索引），避免在每帧更新时遍历整个列表。Area Aura 的目标更新尤其需要注意——500ms 的周期内需要执行一次完整的区域搜索，确保你的空间索引（Grid/Cell 系统）足够高效。
