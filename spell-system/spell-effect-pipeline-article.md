# MMO 法术效果处理管线：完整设计

> 一份面向游戏服务端工程师的法术效果管线设计参考。本文基于对 TrinityCore（World of Warcraft 私有服务器）效果处理系统的深度分析，以纯概念方式描述其架构，使读者能够在自己的项目中复刻类似系统。本文是施法状态机文章的姊妹篇——状态机文章描述"施法如何从发起走向完成"，本文描述"施法完成后效果如何被执行"。

---

## 1 引言

### 1.1 什么是效果处理管线

在 MMO 中，一个"法术"（Spell）可以包含多个"效果"（Effect）。火球术可能同时造成火焰伤害并附带灼烧光环；群体治疗可能同时治疗多个队友并产生过量治疗转化为护盾。

**效果处理管线**（Effect Processing Pipeline）就是将这些效果从"数据定义"变为"实际作用于游戏世界"的执行引擎。它负责：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────────────┐
│  效果数据    │────▶│  数值计算    │────▶│  目标匹配   │────▶│  效果执行    │
│  (配置)     │     │  (修正链)   │     │  (分派)     │     │  (分阶段)    │
└─────────────┘     └──────────────┘     └─────────────┘     └──────────────┘
```

管线的核心设计理念是**数据驱动 + 统一分派**：所有效果类型共享同一条执行管线，通过分派表将效果类型 ID 映射到对应的处理函数。

### 1.2 为什么需要两阶段执行

效果执行不是"算好数值直接扣血"这么简单。考虑以下场景：

- 一个投射物法术（如火球术）需要先"发射"（显示视觉效果、预计算伤害），再等弹道飞到目标后"命中"（实际扣血）
- 施法者在发射阶段和命中阶段之间可能死亡、被沉默、或目标逃跑
- 伤害数值需要在发射时锁定（避免命中时施法者的装备已变化），但命中时需要重新检查目标状态（是否闪避、是否免疫）

因此管线将效果执行分为**两个阶段**：

| 阶段 | 时机 | 职责 |
|------|------|------|
| **发射**（Launch） | 施法完成时立即执行 | 预计算数值、发送视觉效果、锁定暴击结果 |
| **命中**（Hit） | 弹道到达时执行 | 检查命中状态、实际施加效果、触发后续系统 |

这种分离的核心价值是**解耦视觉效果与实际效果的时间**，使投射物法术、延迟法术、引导法术等复杂类型能够在同一条管线中处理。

### 1.3 本文与施法状态机的关系

本文描述的是施法状态机中从"发射状态"到"完成状态"之间的效果执行细节。状态机定义了**什么时候执行**，本文定义了**执行什么、怎么执行**。

```
┌─────────────────────────────────────────────────────┐
│                  施法状态机                          │
│                                                     │
│   初始 ──▶ 准备 ──▶ 发射 ──▶ 完成                    │
│                       │                             │
│              ┌────────┴────────┐                    │
│              │   本文范围       │                    │
│              │   效果处理管线   │                    │
│              └─────────────────┘                    │
└─────────────────────────────────────────────────────┘
```

## 2 四种处理模式

### 2.1 模式定义

管线定义了四种效果处理模式，每种模式对应不同的执行上下文：

| 模式 | 阶段 | 目标上下文 | 典型用途 |
|------|------|-----------|---------|
| **Launch** | 发射阶段 | 无（全局） | 全局预计算、发送视觉效果、统计 |
| **LaunchTarget** | 发射阶段 | 有（逐目标） | 预计算伤害/治疗值、锁定暴击结果 |
| **Hit** | 命中阶段 | 无（全局） | 地面效果创建、区域触发器激活 |
| **HitTarget** | 命中阶段 | 有（逐目标） | 实际施加伤害/治疗/光环 |

"有无目标上下文"的区别在于：无目标模式（Launch/Hit）只能执行不依赖特定目标的逻辑，如创建一个地面效果区域；有目标模式（LaunchTarget/HitTarget）可以访问具体的目标单位并执行定向操作，如扣血、施加光环。

### 2.2 为什么需要四种模式而不是两种

初看似乎"发射"和"命中"两个阶段就够了。但实际的管线需要区分"是否与特定目标交互"，原因如下：

- **全局效果**不需要针对每个目标执行。例如，一个法术创建一个"毒雾区域"——这个效果只需要执行一次（在 Hit 模式下），而不是对区域内每个目标都创建一次
- **数值预计算**需要在发射阶段就针对每个目标计算好，但此时还不能实际施加效果（因为投射物可能还没飞到）。LaunchTarget 模式满足了"有目标但不施加"的需求
- 四种模式形成了**两维分派**：阶段维度（Launch/Hit）× 目标维度（全局/逐目标），效果处理器内部根据当前模式选择性执行对应逻辑

### 2.3 效果处理器的多阶段行为

一个效果处理器可能需要在这四种模式下执行不同逻辑。例如"学校伤害"效果处理器：

```
效果处理器内部逻辑（伪代码）
│
├── 模式 == Launch？
│     └── 发送伤害数值包给客户端（视觉效果）
│
├── 模式 == LaunchTarget？
│     └── 计算伤害值，存入目标记录
│        （计算暴击、计算抗性减免）
│
├── 模式 == Hit？
│     └── 无操作（伤害类效果不需要无目标的 Hit 逻辑）
│
└── 模式 == HitTarget？
      └── 实际扣除目标生命值
```

这种设计让一个效果处理器就能处理完整的生命周期，而不需要为每个阶段创建独立的处理器。

## 3 管线流程总览

### 3.1 完整流程图

效果处理管线的完整流程从法术状态机的"发射"状态开始，到"完成"状态结束：

```
                 cast() / go()
                      │
                      ▼
           ┌──────────────────────┐
           │   有延迟？            │
           │   (发射延迟/弹道飞行) │
           └──────┬───────────────┘
                  │
          ┌───────┴────────┐
          │ NO             │ YES
          ▼                ▼
   ┌────────────┐   ┌──────────────────┐
   │ 即时路径   │   │ 延迟路径          │
   │            │   │                  │
   │ 所有目标   │   │ 等待发射延迟 ──────┤──▶ Launch 阶段
   │ 同步命中   │   │ 等待弹道飞行 ──────┤──▶ 逐目标命中
   │ 一次完成   │   │ 按到达时间分批    │    （未到达的继续等待）
   └─────┬──────┘   └────────┬─────────┘
         │                   │
         └───────┬───────────┘
                 ▼
        ┌─────────────────┐
        │  Launch 阶段    │
        │                 │
        │  1. Launch 模式  │◀── 遍历所有效果（无目标）
        │  2. 目标预处理  │◀── 免疫检查、暴击判定
        │  3. LaunchTarget│◀── 逐目标预计算数值
        └────────┬────────┘
                 ▼
        ┌─────────────────┐
        │  Hit 阶段       │
        │                 │
        │  1. Hit 模式    │◀── 遍历所有效果（无目标）
        │  2. 命中预处理  │◀── 闪避/招架/抵抗判定
        │  3. 效果应用    │◀── HitTarget 模式逐目标
        │  4. 伤害结算    │◀── 实际扣血/治疗
        │  5. 触发系统    │◀── Proc、事件
        └────────┬────────┘

        ┌─────────────────┐
```

### 3.2 分支决策：即时路径 vs 延迟路径

管线的第一个分支点是"是否有延迟"：

- **发射延迟**（Launch Delay）：法术在施法完成后还需要额外等待一段时间才进入发射阶段。常用于"蓄力释放"类技能
- **弹道飞行时间**（Travel Time）：效果需要经过一段飞行时间才命中目标。如火球术从施法者飞向目标需要时间

| 特征 | 即时路径 | 延迟路径 |
|------|---------|---------|
| 触发条件 | 无发射延迟且无弹道飞行时间 | 有发射延迟或有弹道飞行时间 |
| 执行方式 | 一次调用完成全部处理 | 通过事件系统周期性回调，分批处理 |
| 目标命中 | 所有目标同时命中 | 按各自飞行时间分批命中 |
| 状态控制 | 无需额外状态 | 需要标记 Launch/Immediate/目标 是否已处理 |

**设计决策：为什么分离即时和延迟路径？**

延迟路径需要与事件系统集成（定时器回调），这意味着一次处理不能完成所有工作。分离两条路径避免了即时路径承受不必要的事件系统开销。即时路径中所有目标同步命中，逻辑简单且高效。

## 4 Launch 阶段详解

Launch 阶段在发射状态进入后立即执行（即时路径）或在发射延迟结束后执行（延迟路径）。它的核心职责是**预计算**和**视觉效果**。

### 4.1 三步执行

Launch 阶段分为三个有序步骤：

**步骤 1：Launch 模式处理（全局，无目标）**

遍历法术的所有效果，以 Launch 模式调用每个效果处理器。此时所有目标参数为空，效果处理器只能执行全局逻辑。

典型行为：
- 发送法术发射的视觉包给客户端
- 更新全局统计（如连击数计数）
- 触发"法术发射"事件供其他系统监听

**步骤 2：目标预处理（逐目标）**

对每个目标执行发射预处理：
- **免疫性检查**：目标是否对当前法术免疫？如果是，标记为未命中
- **进入战斗判定**：如果施法者和目标属于不同阵营，双方进入战斗状态
- **暴击率计算与判定**：计算暴击概率，投掷随机数，结果存入目标记录。暴击结果在此时锁定，不会在命中阶段重新计算

**步骤 3：LaunchTarget 模式处理（每效果每目标）**

这是 Launch 阶段的核心计算步骤。对每个效果的每个目标：

1. **解析实际命中单位**：考虑反射效果——如果目标有法术反射光环，命中单位变为施法者自身
2. **调用效果处理器**：以 LaunchTarget 模式执行。此时效果处理器计算伤害/治疗值
3. **应用 AoE 衰减**：如果法术是 AoE（区域效果），伤害值会根据目标数量衰减
   - 平方根衰减：有效目标数 = sqrt(实际目标数)
   - 线性上限削减：超过上限的目标受到的伤害线性降低
4. **存储结果**：最终数值存入目标记录，等待 Hit 阶段使用

### 4.2 AoE 伤害衰减机制

AoE 法术对多个目标造成伤害时，如果不加限制，目标越多总伤害越高，可能导致 AoE 法术在群体场景中过于强大。衰减机制确保：

```
目标数量    平方根有效目标    效率
1          1                100%
4          2                50%
9          3                33%
16         4                25%
25         5                20%
```

每个目标实际受到的伤害 = 基础伤害 × 有效目标数 / 实际目标数。

设计决策：**为什么使用平方根衰减而非线性衰减？** 平方根衰减在目标较少时衰减温和（2 个目标效率 71%），在目标较多时衰减显著（10 个目标效率 32%）。这平衡了"小队战斗中 AoE 仍有效"和"大团战中 AoE 不会过强"的需求。

## 5 Hit 阶段详解

Hit 阶段负责**实际的效国施加**。这是效果管线的核心——"让效果真正发生"。

### 5.1 三步执行

Hit 阶段通过统一的"目标容器处理"模板执行，对每个目标执行三个有序步骤：

```
对每个目标：
│
├── 步骤 1：命中预处理（PreprocessTarget）
│     ├── 重新检查目标状态（逃跑？死亡？脱战？）
│     ├── 脚本 BeforeHit 钩子
│     ├── PvP 合法性重检
│     ├── 光环基础值计算
│     ├── 递减效果（DR）计算
│     └── 若 DR 导致持续时间为 0 → 标记为免疫
│
├── 步骤 2：效果应用（DoTargetSpellHit）
│     ├── 创建或刷新光环
│     ├── 设置光环持续时间和 DR 分组
│     └── 以 HitTarget 模式调用效果处理器
│
└── 步骤 3：伤害结算（DoDamageAndTriggers）
      ├── 施加伤害（使用 Launch 阶段预计算的数值）
      ├── 施加治疗
      └── 触发命中后事件（Proc 系统）
```

### 5.2 命中预处理详解

命中预处理是 Hit 阶段的"守门员"，决定效果是否能够实际施加。

**重新检查目标状态**
- 发射时目标存活，但弹道飞行期间目标可能已经逃跑（脱战）、死亡、或转移到了不可达区域
- 重新检查确保在"效果计算"和"效果施加"之间没有状态不一致

**脚本 BeforeHit 钩子**
- 这是脚本系统介入效果处理的最早时机
- 在免疫检查、DR 计算、光环基础值计算尚未执行时触发
- 脚本可以修改目标状态、改变法术行为，甚至阻止后续处理

**递减效果（Diminishing Returns）**
- 控制类效果（昏迷、沉默、减速等）重复施加时会递减：第一次持续 100%，第二次 50%，第三次 25%，之后免疫
- 每个控制效果属于一个 DR 分组，同分组的效果共享递减计数
- 递减计时器在最后一次效果结束后开始重置（通常 18 秒）

**DR 导致持续时间归零的处理**
- 如果递减计算后光环持续时间为 0，且法术仅有光环效果（无伤害等），则整体标记为"免疫"
- 这避免了"施加了一个持续 0 秒的光环"这种无意义操作

### 5.3 效果应用详解

效果应用阶段是"让效果真正发生"的地方。

**光环创建或刷新**
- 检查目标是否已有同类型光环
- 如果有：按堆叠规则决定是刷新持续时间、增加层数、还是被新光环替换
- 如果没有：创建新的光环实例，绑定到目标

**HitTarget 模式效果处理**
- 以 HitTarget 模式调用效果处理器
- 此时效果处理器可以访问完整的目标信息
- 典型行为：实际扣除目标生命值、施加控制效果、消耗目标资源等

### 5.4 伤害结算详解

伤害结算是管线的最后一步"实际生效"操作。

**施加伤害**
- 使用 Launch 阶段在 LaunchTarget 模式中预计算的伤害值
- 为什么不重新计算？因为发射时已经锁定了暴击结果和基础数值，命中时只检查"是否命中"而非"打多少"
- 伤害施加会考虑目标已有的吸收效果（护盾）

**施加治疗**
- 同样使用预计算的治疗值
- 治疗过量部分不会产生负面效果（除非特定机制要求）

**触发命中后事件**
- 命中触发 Proc 系统（如"击中目标有概率触发额外效果"）
- 触发战斗日志系统（记录伤害/治疗数值）
- 触发其他系统的监听器（如成就系统、任务系统）

## 6 数值计算链

### 6.1 计算链总览

效果数值的计算不是简单的"查表取值"。一个法术效果的基础数值需要经过多个修正环节的链式处理：

```
基础值
  │
  ├──▶ 天赋修正（直接设置/乘法修正/不修正）
  │
  ├──▶ 方差随机化（基础值 × 随机浮动范围）
  │
  ├──▶ 资源系数（基础值 × 资源系数，如连击点）
  │
  ├──▶ 等级加成（基础值 + 等级 × 每级加成）
  │
  ├──▶ 连击点加成（基础值 + 连击点 × 单点加成）
  │
  ├──▶ 精通修正（基础值 + 精通值 × 加成系数）
  │
  └──▶ 全局效果修正器（最终调整）
        │
        ▼
      最终数值
```

### 6.2 各环节详解

**基础值（CalcBaseValue）**

基础值的计算方式取决于法术效果的缩放配置：

| 缩放系数 | 计算方式 | 典型场景 |
|---------|---------|---------|
| 非零 | 从缩放曲线查表，按装备等级/法术等级取值 | 大多数现代职业技能 |
| 零 | 直接使用配置表中的固定值 | 旧版技能、固定数值效果 |

缩放曲线的设计使得同一种效果在不同等级时有不同的基础数值，确保"100 级的火球术比 10 级的强"。

**天赋修正**

天赋系统可以修改效果的基础数值，提供三种操作类型：

| 操作类型 | 行为 | 公式 |
|---------|------|------|
| 设置（Set） | 直接覆盖基础值 | value = 天赋值 |
| 乘法（Multiply） | 在基础值上乘以系数 | value = base × 天赋系数 |
| 无（None） | 不修改 | value = base |

**方差随机化**

方差为法术效果引入随机波动，使每次施法的数值不完全相同。例如基础值 100、方差 0.2 的效果，实际值在 80-120 之间随机波动。

计算公式：`value += basePoints × random(-delta, +delta)`，其中 `delta = |variance × 0.5|`。

**资源系数与连击点**

某些效果的基础数值需要乘以资源系数（如盗贼的连击点数量）。连击点越多，终结技的效果越强：

- `comboDamage = value × resourceCoefficient`
- `value += comboDamage × comboPoints`

如果连击点为 5、基础伤害 100、资源系数 0.2：comboDamage = 20，最终 = 100 + 20 × 5 = 200。

**精通修正**

精通是一种角色属性，它对特定效果提供额外加成。精通值越高，对应效果越强：

`value += mastery × bonusCoefficient`

bonusCoefficient 是每个效果独立配置的加成系数。

**全局效果修正器**

这是计算链的最后一环，允许运行时动态调整效果数值。例如某些装备特效会"提升 X% 的火焰伤害"，这就是通过全局效果修正器实现的。

### 6.3 伤害计算的特殊修正

除基础数值链外，伤害还需要额外的修正步骤：

| 修正类型 | 说明 |
|---------|------|
| **暴击** | 伤害 × 暴击倍率（通常 2.0 倍），在 Launch 阶段判定并锁定 |
| **吸收** | 伤害先扣除目标的护盾值，剩余部分才扣除生命值 |
| **减免** | 目标的护甲、抗性等提供百分比减免 |
| **目标乘数** | 某些效果对特定目标有额外的伤害加成或减免 |

设计决策：**为什么暴击在 Launch 阶段判定？** 如果在 Hit 阶段判定暴击，延迟法术在弹道飞行期间施法者可能更换了装备，暴击率会发生变化。在 Launch 阶段锁定确保了"发射时看到什么效果，命中时就造成多少伤害"的一致性。

## 7 效果分派机制

### 7.1 分派表设计

效果分派表是管线的核心路由机制。它将效果类型 ID 映射到对应的处理函数：

```
效果类型 ID                    处理函数
─────────────────────────────────────
   0            ──▶      空操作
   1            ──▶      即时击杀处理器
   2            ──▶      学校伤害处理器
   3            ──▶      虚效果处理器
   ...          ──▶      ...
  354           ──▶      玩家标志处理器
```

分派表是一个固定大小的数组，索引就是效果类型 ID。运行时通过 ID 直接索引数组获取处理函数，时间复杂度为 O(1)。

**设计决策：为什么用函数指针分派表而不是 switch-case？**

| 对比维度 | 函数指针分派表 | switch-case |
|---------|---------------|-------------|
| 查找复杂度 | O(1) 数组索引 | O(log n) 编译器优化后的跳转表 |
| 新增效果 | 在数组对应位置注册 | 在 switch 中添加 case |
| 代码组织 | 处理函数可以独立定义 | 所有逻辑集中在一个大函数中 |
| 可扩展性 | 脚本系统可 hook 任意效果 | 需要在 switch 中插入 hook 点 |
| 空槽位 | 大量 ID 无对应处理器 | 无空槽位开销 |

函数指针分派表的优势在于：查找性能最优、新增效果只需在数组中填入指针、处理函数可以独立维护、脚本系统能通过统一的 hook 点介入任意效果。代价是数组中存在大量空槽位（约 180 个空操作、25 个废弃），但这是以空间换时间的合理权衡。

### 7.2 分派调用流程

从"调用效果处理"到"执行具体逻辑"的完整调用链：

```
HandleEffects()
  │
  ├── 设置当前效果上下文
  │     ├── 当前处理模式（Launch/Hit 等）
  │     ├── 当前目标单位/物品/游戏对象
  │     ├── 当前目标坐标
  │     └── 当前效果信息
  │
  ├── 调用数值计算链
  │     └── 计算当前效果的基础数值（经过修正链处理）
  │
  ├── 调用脚本效果钩子
  │     ├── 根据 mode 选择对应钩子列表
  │     ├── 遍历已注册的脚本处理器
  │     └── 检查是否有脚本阻止了默认行为
  │
  └── 调用分派表处理器
        └── 处理函数指针表[effectID].call()
```

### 7.3 脚本钩子在分派链中的位置

脚本系统在分派表处理器之前执行，这是**最后修改效果行为的机会**：

```
数值计算 ──▶ 脚本钩子 ──▶ 默认处理器
               │
               ├── 脚本可以修改数值
               ├── 脚本可以修改目标
               ├── 脚本可以阻止默认行为（preventDefault）
               └── 脚本可以完全替代默认处理器
```

如果脚本设置了 preventDefault 标志，分派表中的默认处理器将被跳过。这使得脚本系统拥有"完全接管任意效果"的能力。

### 7.4 新增效果类型的标准流程

当游戏设计需要一种新的效果类型时，实现流程如下：

1. **分配效果 ID**：在效果类型枚举中分配一个新 ID
2. **声明处理函数**：在法术类中声明新的效果处理方法
3. **实现处理逻辑**：编写处理函数，内部根据当前处理模式选择性执行：
   - 如果只在 HitTarget 模式下有意义，其他模式直接返回
   - 如果在 Launch 和 Hit 都需要逻辑，用 if 分支处理
4. **注册到分派表**：在分派数组的对应 ID 位置填入函数指针
5. **配置数据**：在效果配置表中为新 ID 定义默认参数
6. **添加脚本支持**（可选）：为新效果注册脚本钩子类型

## 8 效果类型分类

### 8.1 分类总览

一个成熟的 MMO 效果处理系统通常需要覆盖以下效果类别：

| # | 类别 | 核心职责 | 典型效果数 |
|---|------|---------|-----------|
| 1 | 伤害类 | 对目标造成数值伤害 | ~10 |
| 2 | 治疗类 | 恢复目标生命值 | ~6 |
| 3 | 光环类 | 施加/移除/驱散持续效果 | ~8 |
| 4 | 传送与位移 | 改变单位位置（传送、冲锋、击退） | ~15 |
| 5 | 资源与能量 | 汲取/恢复法力、怒气等资源 | ~4 |
| 6 | 召唤类 | 创建实体（宠物、物品、游戏对象） | ~10 |
| 7 | 物品与制造 | 创建物品、附魔、分解 | ~14 |
| 8 | 持久区域 | 创建地面效果、区域触发器 | ~4 |
| 9 | 战斗机制 | 控制、嘲讽、额外攻击 | ~10 |
| 10 | 触发与连锁 | 触发其他法术 | ~8 |

### 8.2 各类别详解

#### 伤害类

伤害类效果是游戏中使用最频繁的效果类型。它们的核心行为模式是统一的：

```
LaunchTarget: 计算伤害值（经过数值计算链）
HitTarget:    施加伤害（扣除目标生命值）
```

变体包括：

| 子类型 | 计算基础 | 特殊行为 |
|--------|---------|---------|
| 学校伤害 | 法术强度 + 基础值 | 受抗性减免 |
| 武器伤害 | 武器 DPS + 基础值 | 受护甲减免 |
| 生命值百分比伤害 | 目标最大生命 × 百分比 | 通常有上限 |
| 环境伤害 | 固定值 | 无施法者归属 |
| 生命汲取 | 计算伤害值 | 伤害的 X% 转化为施法者治疗 |

#### 治疗类

治疗类效果的行为模式与伤害类对称：

```
LaunchTarget: 计算治疗值
HitTarget:    施加治疗（恢复目标生命值）
```

变体包括标准治疗、百分比治疗、最大生命值治疗等。与伤害不同的是，治疗通常不受"抗性"类减免影响。

#### 光环类

光环类效果的核心行为是"在目标上创建一个持续存在的效果修饰器"：

```
HitTarget: 创建/刷新 Aura 实例
```

Aura 的创建涉及复杂的堆叠判定（详见 Aura 架构文章），核心决策包括：
- 是否已有同类型光环？
- 能否与现有光环共存？
- 是刷新持续时间还是增加层数？

#### 传送与位移

位移类效果的核心行为是"改变目标位置"。它们的变体最多，因为游戏中有多种位移模式：

| 位移模式 | 行为 | 典型用途 |
|---------|------|---------|
| 传送 | 立即将单位移到指定位置 | 传送门、回城 |
| 冲锋 | 快速移向目标并触发效果 | 战士冲锋 |
| 击退 | 将目标从施法者推开 | 法师龙息术 |
| 拉取 | 将目标拉向施法者/指定点 | 死亡之握 |
| 跳跃 | 弹道式位移（有抛物线） | 跳跃攻击 |
| 后跳 | 向施法者背后位移 | 后跳 |

这些效果的实现通常涉及服务器端位置插值（确保客户端看到的位移流畅）和碰撞检测。

#### 召唤类

召唤类效果创建新的游戏实体：

- **宠物召唤**：创建一个 AI 控制的友方单位，绑定到施法者
- **物品召唤**：在目标位置创建一个可交互的物品对象
- **游戏对象**：创建一个场景元素（如图腾、传送门）
- **玩家召唤**：将远程的玩家传送到施法者位置

#### 持久区域

持久区域效果创建一个"持续存在的效果区域"：

```
Hit 模式: 创建区域对象，设置持续时间
         区域对象独立运行，周期性对进入区域的目标施加效果
```

与直接 AoE 伤害不同，持久区域是"先放置一个区域，再由区域独立执行效果"。这使得效果可以持续存在很长时间，且区域内的新目标也会受到影响。

#### 触发与连锁

触发类效果的核心行为是"施放另一个法术"：

```
HitTarget: 以当前目标/施法者为上下文，创建并执行新法术
```

变体包括：
- **普通触发**：在当前目标身上施放指定法术。最简单的连锁形式
- **投射物触发**：创建一个飞向目标的投射物，命中后施放指定法术。注意这会产生嵌套的延迟路径——触发的新法术自身可能也有弹道飞行时间
- **强制施法**：无视施法条件（法力、冷却、范围等）强制施放指定法术。用于系统强制行为，如装备特效触发

触发与连锁是构建"技能组合"的基础机制。例如"释放技能 A 后自动触发技能 B"。设计时需要注意**递归触发的深度限制**——如果 A 触发 B，B 又触发 A，会形成无限循环。通常通过"被触发的法术不能再次触发同类效果"的规则来防止。

#### 视觉与场景

视觉效果类效果不直接影响游戏逻辑，但影响玩家体验：

- **播放音效/音乐**：在目标位置播放指定音效
- **播放电影**：触发过场动画
- **创建场景对象**：在指定位置创建一个可见的场景元素
- **创建对话**：在施法者与目标之间启动对话系统

这类效果通常只在 Hit 模式下执行（创建场景对象），不依赖具体目标。

#### 学习与专业技能

学习类效果用于永久性地改变角色状态：

- **学习法术**：将新法术加入角色的可用法术列表
- **学习技能**：提升专业技能等级
- **遗忘专精**：重置角色的天赋配置
- **应用铭文**：修改已有法术的行为

这类效果通常只在 HitTarget 模式下执行，且只对玩家类型的目标有意义。

#### 冷却与充能

冷却类效果用于修改法术的冷却状态，通常被其他法术的触发效果所调用：

- **修改冷却时间**：缩短或延长指定法术的剩余冷却时间
- **批量修改冷却**：一次性修改多个法术的冷却
- **修改充能次数**：直接增加或减少法术的可用充能次数

这类效果是实现"冷却缩减"和"充能恢复"机制的基础。

### 8.3 效果类型的扩展性设计

一个可维护的效果分派系统需要在"类型安全"和"灵活扩展"之间取得平衡：

**ID 分配策略**
- 效果类型 ID 应该是连续的整数枚举，便于数组索引
- 废弃的 ID 保留但映射到空操作，避免 ID 复用导致歧义
- 预留一定范围的 ID 空间给未来扩展

**处理函数的一致性**
- 所有处理函数使用相同的签名（无参数、无返回值，通过上下文传递数据）
- 处理函数内部的第一行应该是模式检查：`if (mode != expected) return;`
- 只在对应模式下执行的逻辑放在模式检查之后

**空操作与废弃**
- 空操作（NULL）处理器应该记录调试日志，便于排查"配置了效果但处理器未实现"的问题
- 废弃处理器可以记录警告信息，提醒策划该效果类型已被替代

## 9 目标信息记录

### 9.1 目标信息层次结构

管线为不同类型的目标提供了统一的信息记录结构：

```
目标信息（基类）
  │
  ├── 单位目标信息
  │     ├── 预计算伤害值
  │     ├── 预计算治疗值
  │     ├── 暴击结果
  │     ├── 未命中原因（闪避/免疫/反射）
  │     ├── 光环创建数据
  │     ├── DR 分组
  │     └── 飞行延迟时间
  │
  ├── 游戏对象目标信息
  │     └── 基本命中判定
  │
  ├── 物品目标信息
  │     └── 基本命中判定
  │
  └── 尸体目标信息
        └── 基本命中判定
```

单位目标信息是最复杂的子类，因为它需要记录完整的"发射-命中"两阶段数据。

### 9.2 效果掩码

每个目标记录中包含一个**效果掩码**（EffectMask），标记哪些效果实际作用于这个目标。

并非所有效果都作用于所有目标。例如：
- 效果 0：对目标 A 施加伤害
- 效果 1：在施法者脚下创建一个区域（与目标 A 无关）
- 效果 2：对目标 A 施加光环

在这种情况下，目标 A 的效果掩码为 `101`（二进制），表示效果 0 和 2 作用于它，效果 1 不作用于它。

效果掩码确保管线只对"真正需要这个效果"的目标执行对应的处理逻辑。

## 10 收尾阶段

### 10.1 完成阶段处理

当所有目标的效果都已处理完毕（即时路径）或所有目标的飞行时间都已到期（延迟路径），管线进入收尾阶段：

```
_handle_finish_phase()
  │
  ├── 额外攻击处理
  │     └── 如果法术触发了额外攻击（如风怒），创建并执行额外攻击
  │
  └── 完成 Proc
        └── 触发"法术完成"事件，供其他系统监听
```

### 10.2 延迟路径的完成判定

延迟路径需要判断"是否所有目标都已处理完毕"。判断方式：

- 维护一个"下一执行时间"变量
- 每次事件回调时，处理所有飞行时间已到期的目标
- 如果所有目标都已处理，"下一执行时间"为 0
- 当"下一执行时间"为 0 时，进入收尾阶段

特殊规则：对于单投射物法术（所有目标共享一个弹道），忽略各目标的飞行时间差异，所有目标同时命中。

### 10.3 延迟路径的状态控制

延迟路径需要额外的状态标志来追踪处理进度：

| 标志 | 用途 | 说明 |
|------|------|------|
| Launch 已处理 | Launch 阶段是否已执行 | 延迟路径首次进入时检查发射延迟，过期后执行 Launch |
| Immediate 已处理 | Immediate Hit 阶段是否已执行 | Launch 后等待延迟过期，过期后执行无目标的 Hit |
| 正在执行中 | 防止重入 | 在事件回调期间标记，防止定时器重复触发 |
| 被引用中 | 防止提前销毁 | 当法术正在被其他系统引用时，禁止销毁 |

这些标志在即时路径中不需要，因为即时路径是一次性完成的，不存在"跨帧持续处理"的场景。但延迟路径的每次事件回调只处理部分目标，因此必须追踪哪些阶段已完成。

### 10.4 防重入与生命周期安全

法术对象在管线执行期间可能面临生命周期问题：

- **施法者死亡**：Launch 之后、Hit 之前施法者可能被杀死。法术效果通常仍会继续执行（已发射的火球不会因为施法者死亡而消失）
- **目标死亡**：弹道飞行期间目标可能死亡。Hit 阶段的命中预处理会检查目标状态
- **法术被引用**：触发效果创建的子法术可能引用父法术的数据。父法术不能在子法术完成前销毁

"正在执行中"标志通过在每次事件回调开始时设置、结束时清除来防止重入。"被引用中"标志通过引用计数确保法术对象在有依赖时不会被释放。

## 11 设计决策回顾

### 11.1 关键决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 执行模型 | 两阶段（Launch + Hit） | 解耦视觉效果与实际效果的时间 |
| 分派机制 | 函数指针分派表 | O(1) 查找、代码解耦、脚本友好 |
| 路径分离 | 即时/延迟双路径 | 避免即时路径承担不必要的事件开销 |
| 数值锁定 | Launch 阶段锁定暴击和基础数值 | 确保发射时的预期与命中时的结果一致 |
| AoE 衰减 | 平方根衰减 | 小队战斗温和、大团战斗显著 |
| 效果扩展 | 统一的四模式处理器 | 一个函数覆盖完整生命周期 |

### 11.2 复刻建议

如果你在自己的项目中实现类似的效果管线，建议的构建顺序：

**第一阶段：最小可用管线**
1. 定义效果类型枚举和分派表（先用 3-5 个效果：伤害、治疗、施加光环）
2. 实现分派调用框架（设置上下文 → 脚本钩子 → 分派表查找 → 执行）
3. 实现最简 HitTarget 模式，让伤害效果能直接生效

**第二阶段：两阶段分离**
4. 添加 Launch 阶段（Launch + LaunchTarget 模式）
5. 将数值计算从 Hit 移到 Launch（LaunchTarget 计算、HitTarget 施加）
6. 添加目标信息记录结构（存储 Launch 阶段计算的结果）

**第三阶段：延迟与复杂化**
7. 添加即时/延迟路径分支
8. 实现事件驱动的延迟路径（定时器 + 逐目标命中）
9. 添加命中预处理（闪避、免疫、DR）

10. 扩展效果类型（传送、召唤、触发等）

**一个关键的早期决策**：先确定你需要的模式数量。如果你的游戏不需要投射物法术（所有效果即时生效），可以简化为"只有 HitTarget 一种模式"，大幅降低系统复杂度。两阶段模型的价值主要体现在有弹道飞行的场景中。
