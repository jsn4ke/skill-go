# MMO 法术脚本扩展系统：完整设计

> 一份面向游戏服务端工程师的法术脚本扩展系统设计参考。本文基于对 TrinityCore（World of Warcraft 私有服务器）脚本系统的深度分析，以纯概念方式描述其架构，使读者能够在自己的项目中复刻类似系统。本文是施法状态机、效果管线、目标选择、Aura 架构文章的姊妹篇。

---

## 1 引言

### 1.1 脚本系统的定位

任何一款运营中的 MMO 都面临一个核心矛盾：**游戏逻辑需要频繁迭代，但核心引擎应该保持稳定**。

每次版本更新，策划团队可能需要新增数十个技能、调整数百个法术的数值和行为。如果这些修改都需要改动引擎代码，那么：

- 引擎代码会急剧膨胀，难以维护
- 每次修改都需要重新编译整个引擎
- 不同技能的逻辑变更会互相干扰

**脚本扩展系统**正是为了解决这个矛盾而存在的。它将引擎定义为"规则的执行者"，将技能的具体行为定义为"脚本"。脚本可以独立于引擎编译、加载、更新，而引擎只负责在正确的时机调用脚本注册的回调。

```
┌─────────────────────────────────────────────────────────────────┐
│                        引擎核心                                  │
│                                                                 │
│   ┌─────────────┐   ┌──────────────┐   ┌──────────────────┐   │
│   │  施法状态机   │──▶│  效果管线     │──▶│  目标选择系统     │   │
│   └─────────────┘   └──────────────┘   └──────────────────┘   │
│          │                  │                   │               │
│          │    在特定阶段调用   │                   │               │
│          ▼                  ▼                   ▼               │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                   脚本扩展系统                            │  │
│   │                                                         │  │
│   │  "法术 12345 命中时，额外造成目标最大生命值 5% 的伤害"      │  │
│   │  "法术 67890 对亡灵目标伤害提高 20%"                      │  │
│   │  "法术 11111 施放前检查目标是否在施法者 40 码内"           │  │
│   └─────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 设计目标

TrinityCore 的脚本扩展系统追求四个核心目标：

**零开销抽象**：脚本回调的调用不应该引入额外的运行时开销（如虚函数表查找）。脚本的注册和调用应当与直接函数调用具有相同的性能特征。

**编译期类型安全**：脚本开发者在注册回调时，如果函数签名不匹配，应该在编译期就报错，而不是等到运行时才发现崩溃。

**多脚本聚合**：同一个法术可以挂载多个脚本。当引擎执行到某个 Hook 点时，所有注册了该 Hook 的脚本都应按序执行。

**双轨体系**：法术分为"瞬态"和"持久态"两种模式，分别由 SpellScript 和 AuraScript 两套脚本体系处理。

### 1.3 与其他子系统的关系

脚本系统不是独立运作的，它嵌入在法术处理的各个阶段中：

```
法术施放时间线
═══════════════════════════════════════════════════════════▶

│ 准备   │ 目标选择 │ 发射  │ Launch │ 计算伤害 │ 命中  │
│        │          │       │        │          │       │
│ 脚本   │ 脚本     │ 脚本  │ 脚本   │ 脚本     │ 脚本  │
│ Hook   │ Hook     │ Hook  │ Hook   │ Hook     │ Hook  │
│ (1-3)  │ (4a-c)   │ (5-6) │ (7-9)  │ (10-11)  │ (12-16)│
```

当法术命中并创建 Aura 时，控制权转交给 AuraScript 体系，处理 Aura 的整个生命周期：

```
Aura 生命周期
═══════════════════════════════════════════════════════════▶

│ 效果应用 │ Tick │ Proc │ 效果移除 │
│          │      │      │          │
│ 脚本     │ 脚本 │ 脚本 │ 脚本     │
│ Hook     │ Hook │ Hook │ Hook     │
```

## 2 脚本注册与加载

### 2.1 注册表架构

脚本系统的入口是一张全局注册表，存储着"法术 ID → 脚本工厂"的映射关系。这张表在编译期就完成了填充。

```
编译期                                      运行时
═════════════════════                        ═════════════════════

脚本文件 A                                     法术施放
├── 注册法术 10001 的脚本 ─┐                      │
└── 注册法术 10002 的脚本 ─┤                      ▼
                          │              ┌──────────────────┐
脚本文件 B                                     │ 查询注册表        │
├── 注册法术 10001 的脚本 ─┤              │ 法术 ID = 10001  │
└── 注册法术 20001 的脚本 ─┤              │ → 找到 2 个脚本  │
                          │              └───────┬──────────┘
                          ▼                      │
                 ┌────────────────┐               ▼
                 │  全局注册表     │      实例化脚本并调用
                 │                │
                 │  10001 → [A,B] │
                 │  10002 → [A]   │
                 │  20001 → [B]   │
                 └────────────────┘
```

**注册表的关键特性**：

- 一个法术可以对应多个脚本（多脚本聚合）
- 注册发生在编译期，通过全局对象的构造函数自动执行
- 注册的是"脚本工厂"（Loader），而非脚本实例本身——脚本实例在法术施放时按需创建

### 2.2 注册流程的三个阶段

脚本的生命周期分为三个截然不同的阶段：

**阶段一：编译期——自动注册**

每个脚本源文件通过一个注册宏声明自己。这个宏在全局作用域创建一个 Loader 对象。由于全局对象的构造函数在程序启动时自动执行，Loader 的构造函数会将自身注册到全局注册表中。

关键点：开发者不需要手动维护任何注册清单。只需在脚本文件末尾写一行注册宏，链接器会确保这个文件被包含进来。

**阶段二：启动期——数据库映射**

服务器启动时，从数据库的 `spell_script` 表中加载映射关系。这张表存储了"法术 ID + 脚本名称"的条目，同时有一个布尔字段控制是否启用。

数据库映射的存在意义在于：它允许在不重新编译的情况下，启用或禁用某个脚本的绑定。这对于运营调试和 A/B 测试非常有用。

```
spell_script 数据库表
┌──────────┬─────────────────────┬────────┐
│ spell_id │ ScriptName          │ enabled│
├──────────┼─────────────────────┼────────┤
│  10001   │ spell_fireball      │   1    │
│  10001   │ spell_fireball_ext  │   1    │
│  10002   │ spell_frostbolt     │   1    │
│  20001   │ spell_shadow_bolt   │   0    │ ← 已禁用
└──────────┴─────────────────────┴────────┘
```

启动时，系统将数据库中的映射关系转换为一个内存中的多重映射表（法术 ID → 脚本名称 + 启用标志）。

**阶段三：运行期——按需实例化**

当法术被施放时，引擎通过以下流程加载脚本：

```
法术施放
    │
    ▼
查询内存映射表，获取该法术 ID 对应的所有脚本条目
    │
    ▼
遍历每个条目
    │
    ├── 检查 enabled 标志 ──→ 未启用则跳过
    │
    ├── 通过脚本名称查找注册表中的 Loader 工厂
    │
    ├── 调用 Loader 创建脚本实例
    │
    ├── 调用脚本的初始化方法（传入法术 ID、脚本名称）
    │
    ├── 调用脚本的加载方法（将脚本与当前 Spell/Aura 对象绑定）
    │
    └── 将脚本实例加入已加载脚本列表
```

脚本实例被创建后，引擎会调用脚本的 `Register` 方法。在这个方法中，脚本通过 `operator+=` 语法将自己关心的回调注册到对应的 Hook 列表中。

### 2.3 脚本的生命周期

一个脚本实例从创建到销毁经历以下状态：

```
 创建实例
    │
    ▼
 ┌─────────────────┐
 │ 注册状态         │  脚本调用 Register()，向 HookList 添加回调
 │ Registration    │
 └────────┬────────┘
          │
          ▼
 ┌─────────────────┐
 │ 加载状态         │  引擎完成所有脚本的加载
 │ Loading         │  回调已就绪，等待执行
 └────────┬────────┘
          │
          ▼
 ┌─────────────────┐
 │ 执行状态         │  法术处理过程中，引擎在各 Hook 点调用回调
 │ Active          │  可访问上下文（目标、伤害值等）
 └────────┬────────┘
          │
          ▼
 ┌─────────────────┐
 │ 卸载状态         │  法术/Aura 结束，清理所有脚本
 │ Unloading       │
 └────────┬────────┘
          │
          ▼
       销毁实例
```

**状态守卫**：系统为每个状态提供了查询方法。当脚本在错误的阶段调用敏感接口时（例如在加载阶段尝试访问命中目标），系统会触发断言失败，帮助开发者尽早发现逻辑错误。

## 3 类型擦除机制

### 3.1 设计目标

脚本系统面临一个根本性的设计难题：**如何存储和调用不同类型的函数指针**。

考虑以下场景：

- 脚本 A 注册了一个命中回调，签名是"无参数，无返回值"
- 脚本 B 注册了一个施放校验回调，签名是"返回施放结果枚举"
- 脚本 C 注册了一个目标选择回调，签名是"接收并修改目标列表"
- 脚本 D 也注册了一个命中回调，但它来自不同的脚本类

引擎需要一个统一的方式存储这些不同签名的回调，并在正确的时机用正确的参数调用它们。

### 3.2 传统方案的问题

**方案一：虚函数**

最直观的方案是让每个脚本类都继承自一个公共基类，基类定义所有可能的虚函数：

```
抽象基类
├── OnBeforeCast()      ── 虚函数
├── OnCheckCast()       ── 虚函数
├── OnEffectLaunch()    ── 虚函数
├── OnHit()             ── 虚函数
├── BeforeHit()         ── 虚函数
├── ...（几十个虚函数）
└── OnAfterHit()        ── 虚函数
```

**问题**：

- 每个脚本对象都携带一个巨大的虚函数表，即使它只重写了一两个方法
- 运行时每次调用都需要通过虚函数表间接寻址，无法内联优化
- 新增 Hook 类型需要修改基类，违反开闭原则

**方案二：类型擦除存储 + 统一分发**

这是 TrinityCore 选择的方案。核心思想是：

- 将任意类型的函数指针"编码"为一个固定大小的字节数组（类型擦除）
- 为每种回调签名生成一个"分发函数"（Thunk），它知道如何从字节数组中恢复原始函数指针并调用

```
┌───────────────────────────────────────────────────────────┐
│                  类型擦除存储单元                            │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  字节缓冲区（固定大小）                              │   │
│  │                                                   │   │
│  │  存储的是原始函数指针的二进制表示                     │   │
│  │  大小 = max(成员函数指针大小, 静态函数指针大小)       │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  分发函数指针（Thunk）                              │   │
│  │                                                   │   │
│  │  指向一个模板静态函数，该函数：                      │   │
│  │  1. 从字节缓冲区恢复原始函数指针                     │   │
│  │  2. 将基类引用安全转型为具体脚本类引用               │   │
│  │  3. 通过原始函数指针调用                             │   │
│  └───────────────────────────────────────────────────┘   │
└───────────────────────────────────────────────────────────┘
```

### 3.3 类型擦除的实现原理

**存储层**：

类型擦除存储单元包含两个部分：

1. **字节缓冲区**：一个固定大小的字节数组，对齐要求与函数指针对齐。其大小取所有可能的函数指针类型中的最大值，确保能容纳任何类型的函数指针。使用对齐修饰保证内存对齐正确。

2. **Thunk 函数指针**：一个统一的函数指针，签名固定为"接收基类引用 + 回调参数 + 字节缓冲区，返回回调结果"。这个 Thunk 是一个模板静态函数的实例化结果。

**构造过程**：

当脚本开发者注册一个回调时，系统执行以下步骤：

```
注册回调: OnHit += HitFn(命中回调方法引用)

1. 确定函数指针类型: 成员函数指针（返回空，无参数）

2. 编译期检查:
   - 缓冲区大小 ≥ 该类型函数指针的大小？ ✓
   - 缓冲区对齐 ≥ 该类型函数指针对齐？ ✓
   - 函数签名与当前 Hook 的预期签名匹配？ ✓

3. 实例化 Thunk:
   - Thunk 指向模板静态函数，该函数知道如何从字节缓冲区中恢复原始成员函数指针

4. 将函数指针写入字节缓冲区（placement new）

5. 存储 Thunk 指针
```

**调用过程**：

当引擎执行到 Hook 点时，系统执行以下步骤：

```
执行 Hook: handler.Call(script)

1. 调用 Thunk(script, args..., buffer)

2. Thunk 内部:
   a. 将字节缓冲区重新解释为 Impl 结构体指针
   b. 从中取出原始函数指针
   c. 将基类引用安全向下转型为具体脚本类引用
   d. 通过原始函数指针调用，传入转型后的引用和参数
```

**编译期签名校验**：

每个 Handler 的构造函数中都包含编译期断言：

- 缓冲区大小必须能容纳实际函数指针类型——如果某种平台的函数指针异常大，编译时会立即报错
- 函数签名必须与 Hook 预期匹配——参数类型错误、返回值类型错误、参数数量错误都会在编译期暴露

这意味着脚本开发者不可能注册一个"接收整数参数"的函数到"不接收参数"的 Hook 中——编译器会直接拒绝。

### 3.4 类型擦除 vs 虚函数对比

| 维度 | 虚函数方案 | 类型擦除方案 |
|------|-----------|-------------|
| 运行时开销 | 虚函数表间接寻址 | 直接函数指针调用，与普通函数调用等价 |
| 内存占用 | 每个对象一个完整虚函数表 | 每个回调仅一个缓冲区 + 一个 Thunk 指针 |
| 内联优化 | 不可能（间接调用） | 理论上可内联（编译器可以看到具体目标） |
| 编译期校验 | 基类定义签名，派生类重写 | Handler 构造函数中的静态断言 |
| 扩展性 | 新增 Hook 需修改基类 | 新增 Hook 只需新增一个 Handler 类 |
| 多回调支持 | 每个虚函数只能有一个实现 | 同一 Hook 可注册多个回调 |
| 编译隔离 | 头文件修改触发大量重编译 | 编译屏障延迟模板实例化 |

**编译屏障**：类型擦除方案还包含一个精妙的编译优化——将 `operator+=` 的模板实例化延迟到编译单元中执行。这意味着当引擎头文件被修改时，数百个脚本文件不需要重新编译模板代码，只在自己对应的编译单元中实例化所需的类型。

### 3.5 类型推导辅助

系统需要从函数指针类型中提取"这个函数属于哪个脚本类"。例如：

- 成员函数指针 → 提取其所属类作为脚本类
- 常量成员函数指针 → 提取其所属类作为脚本类
- 静态函数指针 → 提取其第一个引用参数的类型作为脚本类

这是通过类型特征（type traits）的特化实现的。每种函数指针类型都有对应的提取规则，最终得到脚本类类型。这个类型信息用于在 Thunk 中执行安全的向下转型。

### 3.6 复刻建议

如果你的项目需要实现类似的类型擦除回调系统，建议遵循以下步骤：

1. **定义存储层**：创建一个模板结构体，以函数签名的返回值和参数为模板参数，包含固定大小字节缓冲区和 Thunk 指针
2. **实现构造逻辑**：在 Handler 的模板构造函数中执行静态断言和 placement new
3. **实现调用逻辑**：Thunk 从字节缓冲区恢复函数指针，转型基类引用，调用原始函数
4. **添加编译屏障**：将模板实例化移到编译单元中，避免头文件膨胀
5. **定义 Handler 类族**：为每种 Hook 签名创建一个 Handler 类，复用存储层的模板

## 4 SpellScript Hook 模型

### 4.1 Hook 注册模式

SpellScript 使用一致的注册模式。每个脚本子类必须实现一个 `Register` 方法，在其中通过 `+=` 语法注册回调：

```
class MyFireballScript : SpellScript
    方法 Register():
        BeforeCast += CastFn(施放前回调方法引用)
        OnHit      += HitFn(命中回调方法引用)
        OnEffectHitTarget += EffectFn(
            命中目标效果回调方法引用,
            效果索引 = 0,
            效果类型 = "造成伤害"
        )
```

**效果级 Hook 的过滤**：有些 Hook 只对特定效果索引或效果类型感兴趣。注册时可以指定效果索引（第 0/1/2 个效果）和效果类型（伤害、治疗、施加 Aura 等）。引擎执行时会检查当前效果是否匹配过滤条件，不匹配则跳过。

**宏辅助**：系统为每种 Handler 提供了简洁的宏，开发者不需要直接写 Handler 类型名。宏将方法引用包装为对应类型的 Handler 实例。

### 4.2 主要 Hook 类别与执行时序

SpellScript 定义了 18 个 Hook，按法术处理的阶段严格排序。引擎在法术处理流程中的精确位置调用它们：

**阶段 1：准备（施法条出现前）**

| Hook | 方向 | 用途 |
|------|------|------|
| 法术准备 | 单向通知 | 在施法条开始之前执行自定义逻辑（虚函数覆写，非 HookList） |

**阶段 2：施放校验**

| Hook | 方向 | 用途 |
|------|------|------|
| 施放前 | 单向通知 | 施法条满时、法术处理开始前执行 |
| 施放校验 | 可拦截 | 检查施放条件，可返回失败结果阻止施法 |

**阶段 3：目标选择**

| Hook | 方向 | 用途 |
|------|------|------|
| 区域目标选择 | 可修改 | 筛选区域效果的目标列表（如 AOE 最多命中 5 个目标） |
| 单目标选择 | 可修改 | 替换单个目标（如追踪导弹改为追踪最近目标） |
| 目的地选择 | 可修改 | 修改目标位置（如将落点向施法者方向偏移） |

**阶段 4：发射**

| Hook | 方向 | 用途 |
|------|------|------|
| 法术发射 | 单向通知 | 创建投射物或立即执行前 |
| 发射后 | 单向通知 | 投射物发射后、立即法术完成后 |

**阶段 5：效果发射（Launch 阶段）**

| Hook | 方向 | 用途 |
|------|------|------|
| 效果发射 | 单向通知 | 效果在发射阶段被处理前 |
| 暴击率计算 | 可修改 | 修改暴击率（逐目标计算） |
| 发射目标效果 | 单向通知 | 发射阶段对每个目标执行效果处理 |

**阶段 6：伤害/治疗计算**

| Hook | 方向 | 用途 |
|------|------|------|
| 伤害计算 | 可修改 | 修改最终伤害值（逐目标） |
| 治疗计算 | 可修改 | 修改最终治疗值（逐目标） |
| 抵抗/吸收计算 | 可修改 | 修改伤害抵抗和吸收量（命中前） |

**阶段 7：命中（Hit 阶段）**

| Hook | 方向 | 用途 |
|------|------|------|
| 效果命中 | 单向通知 | 效果在命中阶段被处理 |
| 命中前 | 单向通知 | 单个目标命中前（可查看未命中信息） |
| 命中目标效果 | 单向通知 | 命中阶段对每个目标执行效果处理 |
| 命中 | 可修改 | 伤害结算、Aura 触发前的最终拦截点 |
| 命中后 | 单向通知 | 所有处理完成后的收尾工作 |

**阶段 8：蓄力（10.0 扩展）**

| Hook | 方向 | 用途 |
|------|------|------|
| 蓄力阶段完成 | 单向通知 | 蓄力法术每个阶段蓄满时 |
| 蓄力完成 | 单向通知 | 蓄力法术释放时 |

**特殊 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 成功驱散 | 单向通知 | 效果被成功驱散后触发 |

### 4.3 Hook 方向说明

每个 Hook 的"方向"决定了脚本可以做什么：

- **单向通知**：脚本只能读取上下文信息（如施法者、目标、伤害值），不能改变引擎的后续行为。适用于日志记录、计数统计等场景
- **可修改**：脚本可以修改引擎传递的值（如将伤害值乘以 1.2），修改后的值会被引擎使用
- **可拦截**：脚本可以阻止引擎继续执行（如施放校验返回"目标不在范围内"）

### 4.4 上下文访问控制

不同阶段可以访问不同的上下文信息。这是一个关键的安全设计——脚本不应该在错误的阶段访问尚未准备好的数据：

```
┌───────────────────────────────────────────────────────────┐
│  上下文访问权限矩阵                                         │
├────────────────┬──────────────────────────────────────────┤
│  信息           │  可用阶段                                 │
├────────────────┼──────────────────────────────────────────┤
│  施法者信息     │  所有阶段                                  │
│  法术原型数据   │  所有阶段                                  │
│  效果信息       │  所有阶段                                  │
│  显式目标       │  目标选择完成后                             │
│  目标数量       │  目标选择完成后                             │
│  命中目标       │  命中阶段                                  │
│  命中伤害值     │  命中阶段（可读写）                         │
│  命中治疗值     │  命中阶段（可读写）                         │
│  效果值         │  仅效果处理器中                             │
│  暴击标志       │  命中阶段                                  │
└────────────────┴──────────────────────────────────────────┘
```

如果在错误的阶段调用了受限方法，系统会触发断言失败。例如，在目标选择阶段之前尝试获取"命中目标"就是非法的——此时目标尚未确定。

### 4.5 多脚本执行顺序

当同一个法术注册了多个脚本时，引擎在执行 Hook 时会遍历所有脚本：

```
法术 10001 注册了三个脚本: ScriptA, ScriptB, ScriptC

引擎执行 OnHit Hook:
    for 每个已加载的脚本:
        for 该脚本的 OnHit HookList 中的每个回调:
            执行回调

执行顺序:
    ScriptA.OnHit → ScriptB.OnHit → ScriptC.OnHit
```

**执行顺序保证**：脚本的执行顺序与数据库 `spell_script` 表中的条目顺序一致。数据库中先出现的条目对应的脚本先执行。这为脚本开发者提供了可预测的执行顺序。

**多脚本协作**：多个脚本可以通过修改共享的上下文数据来协作。例如，ScriptA 修改了伤害值，ScriptB 在此基础上再次修改，最终引擎使用的是所有脚本修改后的值。

## 5 AuraScript Hook 模型

### 5.1 与 SpellScript 的衔接

当 SpellScript 的命中阶段执行"施加 Aura"效果时，引擎会创建一个 Aura 对象，然后为这个 Aura 加载 AuraScript。这意味着一个法术可以同时拥有 SpellScript 和 AuraScript——前者处理法术的瞬态行为，后者处理 Aura 的持续行为。

```
法术施放时间线
═══════════════════════════════════════════════════════════▶

         SpellScript 作用域              AuraScript 作用域
    ┌───────────────────┐          ┌──────────────────────┐
    │ 施放 → 目标 → 命中 │          │ 应用 → Tick → Proc  │
    │                   │── Aura ──▶│       → 移除         │
    └───────────────────┘  创建      └──────────────────────┘
```

### 5.2 AuraScript 的两类访问接口

AuraScript 的上下文访问接口分为两类：

**光环级接口**：始终可用，访问 Aura 对象的全局状态。包括：
- 获取法术 ID、施法者、所有者等信息
- 操控持续时间（读取/设置当前持续时间和最大持续时间）
- 操控充能数和层数
- 检查和获取 AuraEffect 指针
- 主动移除整个 Aura

**目标级接口**：仅在涉及特定目标的 Hook 中可用。包括：
- 获取当前正在处理的目标 Unit
- 获取当前目标的 AuraApplication

这种分离的设计是必要的——某些 Hook 是"光环全局"的（如心跳、Proc 检查），而某些 Hook 是"逐目标"的（如效果应用、周期 tick）。在全局 Hook 中尝试访问目标级接口没有意义，系统通过阶段守卫来防止这种误用。

### 5.3 主要 Hook 类别

AuraScript 定义了 28 个 Hook，按功能分为以下类别：

**效果生命周期 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 效果应用时 | 可阻止 | 在 Aura 效果应用到目标之前（可阻止应用） |
| 效果应用后 | 单向通知 | 效果已成功应用到目标 |
| 效果移除时 | 可阻止 | 在效果从目标移除之前（可阻止移除） |
| 效果移除后 | 单向通知 | 效果已从目标移除 |

**周期效果 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 周期效果 Tick | 单向通知 | 周期性效果触发时（如每 3 秒的伤害） |
| 周期效果更新 | 可修改 | 更新周期效果的参数（如修改 Tick 间隔） |

**数值计算 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 效果数值计算 | 可修改 | 计算效果的基础数值（如护盾吸收量） |
| 周期数据计算 | 可修改 | 计算周期效果的频率和数值 |
| 法术修正计算 | 可修改 | 计算法术修正系数（如减少冷却时间） |
| DoT/Hot 暴击率 | 可修改 | 计算持续伤害/治疗的暴击概率 |
| DoT/Hot 伤害/治疗 | 可修改 | 计算每次 Tick 的实际伤害/治疗值 |

**吸收 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 伤害吸收 | 可修改 | 计算伤害吸收量 |
| 伤害吸收后 | 单向通知 | 吸收完成后的处理 |
| 治疗吸收 | 可修改 | 计算治疗吸收量 |
| 治疗吸收后 | 单向通知 | 治疗吸收完成后的处理 |
| 法力盾吸收 | 可修改 | 法力盾的吸收逻辑 |
| 法力盾吸收后 | 单向通知 | 法力盾吸收完成后的处理 |
| 分担伤害 | 可修改 | 伤害分担机制（如灵魂链接） |

**Proc Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 全局 Proc 检查 | 可阻止 | 检查是否满足触发条件 |
| 效果级 Proc 检查 | 可阻止 | 检查特定效果是否满足触发条件 |
| Proc 准备 | 可阻止 | 准备触发前的最终拦截点 |
| Proc 触发 | 单向通知 | Proc 已触发，执行效果 |
| Proc 触发后 | 单向通知 | Proc 效果执行完成 |
| 效果级 Proc | 单向通知 | 特定效果的 Proc 触发 |
| 效果级 Proc 后 | 单向通知 | 特定效果的 Proc 处理完成 |

**其他 Hook**

| Hook | 方向 | 用途 |
|------|------|------|
| 区域目标检查 | 可阻止 | 区域 Aura 对新目标的准入检查 |
| 驱散 | 单向通知 | Aura 被驱散时 |
| 驱散后 | 单向通知 | 驱散完成后 |
| 心跳 | 单向通知 | 单位每次心跳时触发 |
| 进出战斗 | 单向通知 | 目标进入或离开战斗时 |

### 5.4 AuraScript 的状态栈机制

AuraScript 面临一个 SpellScript 不存在的问题：**Hook 的嵌套调用**。

一个 Aura 可能包含多个效果，每个效果都有自己的生命周期 Hook。当引擎处理效果 A 的应用 Hook 时，可能会触发效果 B 的某些逻辑，导致效果 B 的 Hook 被调用。如果不做特殊处理，效果 B 的 Hook 中的"阻止默认行为"操作会影响到效果 A。

AuraScript 使用**状态栈**解决这个问题：

```
效果 A 应用 Hook 开始
    │
    ├── 推入状态栈: {效果A的 AuraApplication, 当前Hook类型, 默认行为未被阻止}
    │
    ├── 执行脚本回调
    │       │
    │       └── 触发效果 B 的逻辑
    │               │
    │               ├── 推入状态栈: {效果B的 AuraApplication, 当前Hook类型, 默认行为未被阻止}
    │               │
    │               ├── 执行效果 B 的回调（可能阻止默认行为）
    │               │
    │               └── 弹出状态栈 → 恢复效果 A 的状态
    │
    ├── 弹出状态栈 → 检查效果 A 的阻止标志
    │
    └── 根据效果 A 的阻止标志决定是否执行默认行为
```

**状态栈的每个条目保存**：

1. 当前正在处理的 AuraApplication（确定是哪个目标）
2. 当前 Hook 类型（确定是哪个阶段）
3. 默认行为是否已被阻止（布尔标志）

每次 Hook 调用前推入栈，调用后弹出栈。这保证了：

- 内层 Hook 的阻止操作不会影响外层 Hook
- 每个目标的效果处理是独立的
- 嵌套调用结束后状态自动恢复

## 6 阻止默认行为机制

### 6.1 设计目标

脚本系统不仅需要"在正确时机执行自定义逻辑"，还需要"阻止引擎的默认行为"。这是实现特殊技能效果的关键能力。

考虑以下场景：

- 一个技能"不造成伤害，但施加一个 Debuff"——需要阻止默认的伤害执行
- 一个 Buff"不可被驱散"——需要在驱散检查时阻止默认的移除行为
- 一个 Proc 效果"只在暴击时触发"——需要在 Proc 检查中阻止非暴击情况的默认触发

### 6.2 SpellScript 的阻止机制

SpellScript 使用两个**位掩码**来追踪被阻止的效果：

```
阻止级别:

位掩码 1: 完全阻止
    阻止指定效果的全部执行
    包括其他脚本对该效果的回调
    例如: PreventHitEffect(效果0) → 效果0 的所有处理都被跳过

位掩码 2: 阻止默认行为
    仅阻止引擎对指定效果的默认处理
    脚本的回调仍然可以运行
    例如: PreventHitDefaultEffect(效果0) → 效果0 的引擎默认处理被跳过，但脚本回调正常执行
```

**额外的便捷阻止方法**：

- 阻止命中伤害：等价于将伤害值设为 0
- 阻止命中治疗：等价于将治疗值设为 0
- 阻止命中 Aura：阻止在当前目标上应用 Aura

**内部检查**：引擎在执行效果之前会检查位掩码，如果对应位被设置，则跳过该效果的处理。这确保了阻止操作不会遗漏已经处理过的效果——只有尚未处理的效果可以被阻止。

### 6.3 AuraScript 的阻止机制

AuraScript 的阻止机制与 SpellScript 不同，它使用一个**布尔标志**配合**状态栈**：

```
阻止流程:

1. Hook 调用开始前: 重置 m_defaultActionPrevented = false

2. 脚本回调执行:
   - 如果脚本调用 PreventDefaultAction()
   - 则设置 m_defaultActionPrevented = true

3. Hook 调用结束后: 引擎检查 m_defaultActionPrevented
   - 如果为 true → 跳过默认行为
   - 如果为 false → 正常执行默认行为
```

**适用范围**：并非所有 Hook 都支持阻止默认行为。只有明确标记为"可阻止"的 Hook（如效果应用时、效果移除时、Proc 检查等）才响应阻止操作。在不可阻止的 Hook 中调用阻止方法不会产生任何效果。

### 6.4 阻止机制的设计理由

**为什么 SpellScript 使用位掩码，而 AuraScript 使用布尔标志？**

SpellScript 处理的是"一次性"的命中流程——多个效果依次处理，每个效果都可以独立被阻止。因此位掩码（每个效果一个位）是自然的表达方式。

AuraScript 处理的是"持续性"的生命周期——每次 Hook 调用是独立的，不需要跨调用保持阻止状态。因此布尔标志配合状态栈更加简洁。

**为什么需要区分"完全阻止"和"阻止默认行为"？**

"完全阻止"是强力的——它会阻止一切，包括其他脚本的回调。适用于"这个效果完全不应该存在"的场景。

"阻止默认行为"是温和的——引擎的默认处理被跳过，但脚本仍然可以执行自定义逻辑。适用于"用自定义行为替换默认行为"的场景。

这种分级设计给脚本开发者提供了精细的控制能力。

## 7 跨子系统协作

### 7.1 脚本系统与其他子系统的接口

脚本系统不是孤立存在的，它通过上下文访问接口与引擎的各个子系统交互：

```
┌───────────────────────────────────────────────────────────┐
│                     脚本系统                                │
│                                                           │
│  ┌─────────────┐  ┌──────────────┐  ┌──────────────────┐ │
│  │ SpellScript  │  │  AuraScript   │  │  共享基础设施    │ │
│  │             │  │              │  │                  │ │
│  │ 18 个 Hook   │  │ 28 个 Hook    │  │ 类型擦除存储     │ │
│  │ 上下文访问   │  │ 上下文访问    │  │ HookList 聚合   │ │
│  │ 阻止机制     │  │ 状态栈       │  │ 效果过滤基类     │ │
│  └──────┬──────┘  └──────┬───────┘  └──────────────────┘ │
│         │                │                                │
└─────────┼────────────────┼────────────────────────────────┘
          │                │
    ┌─────▼──────┐   ┌────▼───────┐
    │  法术系统    │   │  Aura 系统  │
    │             │   │            │
    │ 施法状态机   │   │ 三层架构    │
    │ 效果管线     │   │ 堆叠逻辑    │
    │ 目标选择     │   │ Proc 管线  │
    └─────────────┘   └────────────┘
```

### 7.2 脚本拦截点总览

脚本可以在法术处理的几乎每个阶段进行拦截。以下是关键的拦截点及其对应的引擎逻辑：

**在施法前拦截**：
- 检查施放条件（目标是否在范围内、施法者是否有足够资源）
- 返回失败结果可以完全阻止法术施放
- 替代方案：修改引擎的施放条件数据库表

**在目标选择时拦截**：
- 筛选目标列表（移除免疫目标、限制目标数量）
- 替换单个目标（追踪技能改为追踪特定目标）
- 替代方案：修改目标选择算法的数据库参数

**在命中时拦截**：
- 修改伤害/治疗值
- 阻止默认效果执行
- 替代方案：无——这是唯一能在最终数值确定前进行拦截的点

**在 Aura 生命周期中拦截**：
- 控制效果的应用和移除
- 计算周期效果的数值
- 拦截 Proc 触发
- 替代方案：无——Aura 的行为高度依赖于脚本定制

## 8 复刻指南

### 8.1 关键设计决策回顾

| 决策 | 选择 | 理由 |
|------|------|------|
| 回调存储 | 类型擦除 + 字节缓冲区 | 零运行时开销，编译期类型安全 |
| 多脚本聚合 | HookList 容器 + 顺序遍历 | 简单可预测，支持多脚本协作 |
| Hook 执行顺序 | 数据库表条目顺序 | 可配置，无需重新编译 |
| 阻止机制 | 位掩码（Spell）+ 布尔标志（Aura） | 匹配各自的语义需求 |
| 状态安全 | 阶段守卫 + 断言 | 编译期无法覆盖的场景用运行时检查 |
| 嵌套调用安全 | 状态栈（AuraScript） | 防止内层 Hook 影响外层 Hook |
| 编译隔离 | 编译屏障延迟模板实例化 | 避免头文件修改触发大规模重编译 |
| 效果过滤 | EffectHook 基类 | 统一效果索引/类型过滤逻辑 |

### 8.2 实现步骤建议

**第一步：定义类型擦除存储层**

这是整个系统的基础。创建一个模板结构体，接受返回值类型和参数类型作为模板参数。内部包含固定大小字节缓冲区和 Thunk 函数指针。确保缓冲区大小取所有可能函数指针类型中的最大值。

**第二步：定义 Handler 类族**

为每种 Hook 签名创建一个 Handler 类。每个 Handler 的模板构造函数中执行编译期断言（大小、对齐、签名匹配），然后将函数指针写入缓冲区，设置 Thunk 指针。

**第三步：实现 HookList 容器**

创建一个简单的向量包装器，提供 `operator+=` 接口。如果编译时间是大问题，添加编译屏障延迟模板实例化。

**第四步：定义脚本基类**

创建 SpellScript 和 AuraScript 两个基类，各自持有所有 Hook 类型的 HookList 成员。定义 `Register` 纯虚函数。

**第五步：实现注册表**

创建全局注册表，支持通过宏在编译期自动注册脚本工厂。实现数据库映射加载，支持启用/禁用控制。

**第六步：实现加载流程**

在法术/Aura 创建时查询注册表，实例化脚本，调用 `Register`，收集所有回调。

**第七步：实现上下文访问接口**

为每个阶段提供可用的查询方法和修改方法。添加阶段守卫，在错误调用时触发断言。

**第八步：实现阻止机制**

SpellScript 使用位掩码，AuraScript 使用布尔标志配合状态栈。引擎在执行默认行为前检查阻止标志。

### 8.3 常见陷阱

**陷阱一：在错误阶段访问上下文**

脚本开发者经常在效果发射阶段尝试获取命中目标。此时命中尚未发生，目标尚未确定。系统会触发断言，但这个错误在运行时才会暴露。

建议：为每种 Hook 编写文档，明确列出可用的上下文方法。

**陷阱二：多脚本修改顺序**

多个脚本修改同一个值（如伤害值）时，后执行的脚本会看到前一个脚本的修改结果。如果脚本 B 期望的是原始伤害值而非脚本 A 修改后的值，就会产生错误。

建议：在脚本设计时明确依赖关系，通过数据库表中的条目顺序控制执行顺序。

**陷阱三：阻止行为的理解偏差**

"完全阻止"和"阻止默认行为"的区别容易被忽略。在需要保留脚本自定义逻辑的场景中错误使用了"完全阻止"，会导致所有脚本回调都被跳过。

建议：默认使用"阻止默认行为"，只在确实需要完全跳过时使用"完全阻止"。

---

## 9 总结

TrinityCore 的法术脚本扩展系统是一个精心设计的回调框架，其核心创新在于类型擦除机制。通过将不同类型的函数指针统一存储为固定大小的字节缓冲区，系统实现了零运行时开销的类型安全回调调用。

这个系统的设计体现了几个重要的工程原则：

**关注点分离**：引擎负责定义"何时"（Hook 点），脚本负责定义"做什么"（回调逻辑）。两者通过清晰的接口契约连接。

**编译期保证**：尽可能多地将错误检测前移到编译期。函数签名不匹配、缓冲区大小不足等问题在编译时就暴露，而不是等到运行时才发现。

**渐进式安全**：编译期无法覆盖的场景（如在错误阶段访问上下文）通过运行时断言兜底。既不牺牲安全性，也不引入不必要的开销。

**可扩展性**：新增 Hook 类型只需添加一个新的 Handler 类和对应的 HookList 成员，不需要修改基类或已有的脚本代码。

对于希望在自己的项目中复刻类似系统的工程师，建议从类型擦除存储层开始实现，这是整个系统最核心也最有价值的部分。其余的注册表、加载流程、上下文接口等都可以在这个基础上逐步构建。
