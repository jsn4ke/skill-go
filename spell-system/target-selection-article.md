# MMO 法术目标选择系统：完整设计

> 一份面向游戏服务端工程师的目标选择系统设计参考。本文基于对 TrinityCore（World of Warcraft 私有服务器）目标选择系统的深度分析，以纯概念方式描述其架构，使读者能够在自己的项目中复刻类似系统。本文是施法状态机、效果管线文章的姊妹篇——状态机文章描述"施法如何流转"，管线文章描述"效果如何执行"，本文描述"效果作用于谁"。

---

## 1 引言

### 1.1 什么是目标选择系统

当玩家释放一个群体治疗法术时，服务器需要回答一个问题：**这个治疗应该作用于哪些队友？** 当一个火球术飞向目标时，服务器需要确认：**目标是否仍然有效？** 当一个锥形范围的横扫攻击发动时，服务器需要确定：**哪些敌人处于锥形范围内？**

目标选择系统（Target Selection System）就是回答"法术作用于谁"的子系统。它的核心职责是：

- 从客户端请求和法术配置中确定所有受影响的目标
- 根据空间关系（距离、方向、形状）筛选目标
- 根据游戏规则（阵营、状态、条件）过滤目标
- 将最终的目标列表交付给效果处理管线

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  客户端请求  │────▶│  显式目标    │────▶│  隐式目标    │────▶│  合并去重    │
│  (玩家指定)  │     │  (解析验证)  │     │  (规则扩展)  │     │  (最终列表)  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
                                                                   │
                                                                   ▼
                                                            ┌──────────────┐
                                                            │  效果管线    │
                                                            │  (逐目标执行)│
                                                            └──────────────┘
```

### 1.2 为什么目标选择如此复杂

目标选择的复杂性来自三个维度的交叉：

- **来源多样**：目标可能来自玩家手动选择、法术自身定义的选取规则、前一个法术的命中结果（引导法术继承目标）
- **规则多样**：不同法术需要不同的空间形状（圆形区域、锥形、直线）、不同的过滤条件（敌方、友方、特定类型）、不同的参考点（以施法者为中心、以目标为中心）
- **可定制需求**：脚本系统需要能够拦截和修改目标列表，实现特殊的法术行为

如果用传统的嵌套条件来处理，代码会迅速膨胀为不可维护的状态。因此系统需要一个**声明式的、可组合的目标描述框架**。

## 2 声明式目标描述

### 2.1 核心设计理念

传统方式下，每种法术的目标选取逻辑通过硬编码实现：

```
火球术:  target = GetExplicitTarget()
群体治疗: targets = GetAlliesInRange(caster, 40)
锥形攻击: targets = GetEnemiesInCone(caster, 10, 90度)
闪电链:  target = GetExplicitTarget(); chain = GetChainTargets(target, 3, 10)
```

每种新法术都需要编写自定义的目标选取逻辑。当法术数量达到数千种时，维护成本急剧上升。

TrinityCore 的解决方案是**声明式目标描述**：每个法术效果通过两个目标字段（通常称为 TargetA 和 TargetB）声明目标选取方式。运行时引擎根据这些声明式描述自动完成目标选取，无需为每个法术编写特定的逻辑。

声明式描述的关键在于：**将目标选取行为参数化为一个多维参数空间**，使上百种目标类型统一到少数几个标准化的选取算法中。

### 2.2 五维度分解

每个目标声明被映射到一个**五维参数空间**，五个维度共同定义了一种完整的目标选取行为：

| 维度 | 名称 | 回答的问题 | 类比 |
|------|------|-----------|------|
| **SelectionCategory** | 选择类别 | 用什么算法选取？ | "搜索方式" |
| **ReferenceType** | 参考系 | 以谁为参考？ | "坐标原点" |
| **ObjectType** | 对象类型 | 选取什么类型的对象？ | "目标种类" |
| **CheckType** | 验证类型 | 选取条件是什么？ | "过滤规则" |
| **DirectionType** | 方向类型 | 朝哪个方向？ | "朝向角度" |

五个维度的组合可以描述任意一种目标选取行为。例如：

- "在施法者周围 10 码范围内找所有敌方单位" → Area + Caster + Unit + Enemy + None
- "在目标身后 5 码范围内找所有友方单位" → Area + Target + Unit + Ally + Back
- "以施法者前方 45 度角、8 码半径的锥形范围内找敌人" → Cone + Caster + Unit + Enemy + FrontRight

**设计决策：为什么用 5 维度分解而不是嵌套 if-else？**

| 对比维度 | 5 维度分解 | 嵌套 if-else |
|---------|-----------|-------------|
| 可扩展性 | 新增目标类型只需在查找表中添加一行 | 每种类型需要新的 if 分支 |
| 可组合性 | 5 个维度正交组合，2×6×11×10×11 = 14,520 种组合 | 组合需要深层嵌套 |
| 数据驱动 | 配置表即可定义新法术的目标行为 | 需要修改代码 |
| 可维护性 | 查找表集中管理，一目了然 | 逻辑分散在多个函数中 |
| 脚本友好 | 脚本可以读取维度信息做决策 | 脚本无法理解硬编码逻辑 |

5 维度分解的代价是需要维护一张静态映射表（将原始枚举 ID 映射到五维参数）。但这是一次性工作，且表的格式远比散落的 if-else 容易审阅。

## 3 五个维度详解

### 3.1 维度一：SelectionCategory（选择类别）

选择类别决定了使用哪种**空间搜索算法**来选取目标。

| 值 | 算法 | 典型用途 |
|----|------|---------|
| **默认** | 无空间搜索，直接指定对象或位置 | 单体法术（如单体治疗、单体攻击） |
| **引导** | 从当前引导法术中继承目标 | 引导法术的目标延续 |
| **附近** | 在参考点附近搜索单个目标 | 智能治疗（自动选择最近的受伤队友） |
| **锥形** | 以参考点为中心、指定角度的锥形范围搜索 | 前方横扫、冰锥术 |
| **区域** | 以参考点为中心的圆形范围搜索 | 群体治疗、暴风雪 |
| **弹道** | 沿抛物线路径检测碰撞 | 投石、箭雨 |
| **直线** | 以参考点到终点的直线为轴的矩形范围搜索 | 直线 AOE、穿越法术 |

**为什么需要 7 种算法而不是用一种通用算法？** 因为不同算法的空间复杂度和性能特征差异巨大。附近搜索只需找到第一个匹配目标（可以提前终止），而区域搜索需要遍历所有匹配目标。锥形搜索需要角度计算，弹道搜索需要碰撞检测。使用专用算法可以在各自场景下达到最优性能。

### 3.2 维度二：ReferenceType（参考系）

参考系决定了目标选取的**空间参考点**——"以谁为中心进行搜索"。

| 值 | 参考对象 | 说明 |
|----|---------|------|
| **无** | 无参考 | 不需要空间参考（默认类别） |
| **施法者** | 施法者自身 | 以施法者位置为中心搜索 |
| **目标** | 显式目标单位 | 以玩家选中的目标为中心搜索 |
| **最后** | 最后添加的目标 | 以上一个效果命中过的目标为中心 |
| **源位置** | 源坐标 | 以法术的起始坐标为中心搜索 |
| **目标位置** | 目标坐标 | 以法术的终止坐标为中心搜索 |

参考系的灵活组合是系统表达力的核心。例如：

- "在施法者脚下释放 AOE" → 参考系 = 施法者
- "在目标脚下释放 AOE" → 参考系 = 目标
- "在施法者到目标的路径上搜索" → 参考系 = 源位置/目标位置

**"最后"参考系的特殊价值**：当一个法术有多个效果时，效果 B 可能需要在效果 A 命中的目标附近搜索。例如"对目标造成伤害，然后在目标周围释放一个 AOE"——此时 AOE 的参考系就是"效果 A 最后命中的目标"。

### 3.3 维度三：ObjectType（对象类型）

对象类型决定了目标选取返回的**世界对象种类**。

| 值 | 返回类型 | 说明 |
|----|---------|------|
| **无** | 无 | 不返回对象目标 |
| **源位置** | 坐标 | 设置法术的源位置 |
| **目标位置** | 坐标 | 设置法术的目标位置 |
| **单位** | 游戏单位 | 玩家、NPC、宠物等 |
| **单位+位置** | 单位+坐标 | 同时返回单位和其位置 |
| **游戏对象** | 游戏对象 | 宝箱、门、矿脉等可交互物体 |
| **物品** | 物品 | 背包中的物品 |
| **尸体** | 尸体 | 友方或敌方的尸体 |
| **敌方尸体** | 尸体 | 仅敌方尸体（用于复活、拾取） |
| **友方尸体** | 尸体 | 仅友方尸体（用于复活） |

对象类型决定了搜索的"容器"——在世界对象管理系统中，单位、游戏对象、物品存储在不同的容器中。指定对象类型可以让搜索只在对应的容器中执行，避免不必要的遍历。

### 3.4 维度四：CheckType（验证类型）

验证类型决定了目标选取的**过滤条件**——"什么样的目标才符合要求"。

| 值 | 过滤条件 | 典型用途 |
|----|---------|---------|
| **默认** | 通用检查 | 根据法术正负性自动选择 |
| **条目** | 按 Entry 过滤 | 只选择特定类型的生物/对象 |
| **敌方** | 仅敌方 | 攻击类法术 |
| **友方** | 仅友方 | 治疗/增益类法术 |
| **队伍** | 仅队伍成员 | 小队治疗 |
| **团队** | 仅团队副本成员 | 团队治疗 |
| **团队职业** | 团队中指定职业 | 特定职业增益 |
| **乘客** | 载具乘客 | 载具相关法术 |
| **召唤物** | 仅召唤物 | 召唤物管理法术 |

**验证类型是目标选取的"守门员"**：它决定了哪些对象可以被选中，哪些会被排除。大多数法术只需要"敌方"或"友方"两种基本过滤，但复杂场景需要更细粒度的控制。

**"条目"验证的特殊之处**：它允许按生物/对象的唯一标识（Entry）进行过滤，配合条件系统可以实现非常精确的目标选择。例如"只对名为 XXX 的生物生效"或"只对特定类型的矿脉生效"。

### 3.5 维度五：DirectionType（方向类型）

方向类型决定了目标选取的**朝向角度**，主要用于锥形范围和目标位置偏移。

| 值 | 角度 | 方向 |
|----|------|------|
| **无** | - | 不需要方向 |
| **正前方** | 0° | 施法者面向方向 |
| **正后方** | 180° | 施法者背对方向 |
| **右侧** | -90° | 施法者右侧 |
| **左侧** | 90° | 施法者左侧 |
| **右前方** | -45° | 施法者右前方 |
| **右后方** | -135° | 施法者右后方 |
| **左后方** | 135° | 施法者左后方 |
| **左前方** | 45° | 施法者左前方 |
| **随机** | 随机 | 随机方向 |
| **条目** | 由条件决定 | 根据目标条件动态确定 |

方向类型主要用于两种场景：

1. **锥形搜索**：锥形需要知道"朝哪个方向展开"。正前方的锥形是前方横扫，正后方的锥形是背后攻击
2. **位置偏移**：某些法术需要在参考点的某个方向偏移一定距离。例如"在目标左后方 5 码处创建效果"

### 3.6 维度组合示例

五维度的组合可以用一句话描述任何目标选取行为：

| 组合 | 含义 | 典型法术 |
|------|------|---------|
| Area + Caster + Unit + Enemy + None | 在施法者周围找敌方单位 | 奉献（圣骑士 AOE） |
| Area + Target + Unit + Ally + None | 在目标周围找友方单位 | 群体治疗 |
| Cone + Caster + Unit + Enemy + Front | 施法者前方锥形范围内找敌人 | 雷霆一击 |
| Line + Dest + Unit + Enemy + None | 从施法者到目标位置的直线上找敌人 | 英勇打击 |
| Default + Caster + Unit + Ally + None | 直接选中施法者自身 | 自我治疗 |
| Area + Caster + Unit + Raid + None | 在施法者周围找团队成员 | 祷言 |
| Nearby + Caster + Unit + Ally + None | 在施法者附近找单个友方单位 | 智能治疗 |
| Cone + Caster + Unit + Enemy + FrontRight | 施法者右前方 45° 锥形范围 | 特殊方向攻击 |
| Traj + Dest + Dest + Default + None | 沿弹道路径检测碰撞并调整落点 | 投石 |

## 4 显式目标与隐式目标

### 4.1 显式目标

**显式目标**（Explicit Target）是玩家在客户端明确选择的目标——点击一个敌人、选中一个友方、或者指定一个地面位置。

显式目标通过客户端-服务器的通信协议传递，包含以下信息：

```
显式目标数据
│
├── 对象目标
│     ├── 单位目标（点击的玩家/NPC）
│     ├── 物品目标（选中的物品）
│     └── 游戏对象目标（选中的宝箱/矿脉）
│
├── 位置目标
│     ├── 源位置（法术起始点）
│     └── 目标位置（法术终止点/地面标记点）
│
└── 弹道参数
      ├── 仰角（抛射物角度）
      └── 速度（飞行速度）
```

**显式目标的解析流程**

1. **接收数据**：从客户端发来的施法请求中提取目标信息
2. **验证匹配**：检查客户端提供的对象目标是否与法术需求匹配（法术需要单位目标但客户端没有提供）
3. **自动补充**：如果客户端没有提供必需的目标，尝试自动选择：
   - 玩家施法器：使用当前选中的目标
   - NPC 施法器：使用当前攻击目标
   - 兜底：若法术需要友方目标，使用施法者自身
4. **位置补全**：若法术需要地面位置但未提供，使用对象目标的位置或施法者位置
5. **目标重定向**：检查是否存在重定向效果（如接地图腾将有害法术重定向到图腾）

**目标重定向机制**

目标重定向是显式目标处理中的"安全网"。典型场景：

```
玩家对术士施放有害法术
      │
      ▼
检查：术士身旁是否有接地图腾？
      │
      ├── YES → 将法术目标重定向到图腾上
      │
      └── NO  → 目标不变
```

重定向根据法术的伤害类型（魔法/近战/远程）选择不同的重定向逻辑。这种机制使得接地图腾成为一个"法术吸收器"——任何有害法术都会被自动导向图腾，无需为每个法术单独编写重定向逻辑。

**目标类型标志**

客户端在施法请求中通过位标志（bitmask）描述包含的目标类型。服务器端根据法术的配置需求验证这些标志是否匹配：

| 标志类别 | 包含的标志 | 说明 |
|---------|-----------|------|
| 单位类 | 普通单位、团队成员、队伍成员、敌方、友方 | 覆盖所有活着的单位 |
| 游戏对象类 | 普通游戏对象、带物品的游戏对象 | 覆盖可交互对象 |
| 尸体类 | 友方尸体、敌方尸体 | 覆盖已死亡单位 |
| 物品类 | 交易物品、背包物品 | 覆盖物品目标 |

标志的验证确保客户端发送的目标类型与法术需求一致。例如，如果法术需要"敌方单位"但客户端发送的是"友方单位"，目标会被拒绝。

**自动目标选择策略**

当客户端没有提供目标、或提供的目标不合法时，系统按以下优先级尝试自动选择：

```
1. 使用施法者当前选中的目标
      └── 如果是 NPC 施法器，使用当前攻击目标

2. 如果法术需要友方目标：
      └── 使用施法者自身

3. 如果法术需要地面位置但未提供：
      └── 使用对象目标的位置（如果有）或施法者位置

4. 如果法术需要源位置但未提供：
      └── 使用施法者位置

5. 如果以上全部失败：
      └── 以"需要目标"错误拒绝施法
```

### 4.2 隐式目标

**隐式目标**（Implicit Target）是法术效果自身定义的目标选取规则，存储在法术的配置数据中。

隐式目标的处理流程：

```
对每个效果（一个法术最多 3 个效果，每个效果有 TargetA 和 TargetB）:
│
├── 读取 TargetA 的五维参数
│     ├── SelectionCategory → 决定搜索算法
│     ├── ReferenceType → 决定参考点
│     ├── ObjectType → 决定目标种类
│     ├── CheckType → 决定过滤条件
│     └── DirectionType → 决定方向
│
├── 读取 TargetB 的五维参数
│     └── （同上）
│
├── 执行 TargetA 的目标搜索
│     └── 根据搜索算法（Area/Cone/Line/Nearby 等）选取目标
│
├── 执行 TargetB 的目标搜索
│     └── （同上）
│
└── 将所有搜索结果合并到统一的目标列表
```

### 4.3 TargetA 与 TargetB 的协作

每个效果有两个目标字段（TargetA 和 TargetB），它们独立处理，最终结果合并到同一个目标列表。这种设计的价值在于：

- **分离关注点**：TargetA 描述"主要目标"，TargetB 描述"次要目标"或"位置效果"
- **效果复用**：同一个效果定义可以被不同的法术使用，只需更改 TargetA/B 的值
- **组合灵活**：通过 A+B 的组合可以实现复杂的目标行为

常见的 TargetA + TargetB 组合模式：

| TargetA | TargetB | 含义 | 举例 |
|---------|---------|------|------|
| 单位目标 | 目标位置 | 对目标施加效果 + 在目标位置产生效果 | 施加 DOT 并在脚下放毒雾 |
| 区域 A | 区域 B | 两层区域效果共享搜索结果 | 一个法术同时治疗和净化 |
| 单体 | 链式 | 先选定目标，再链式跳跃到额外目标 | 闪电链 |
| 引导继承 | 区域 | 继承引导目标 + 在目标周围搜索 | 引导期间周期性 AOE |
| 地面位置 | 弹道 | 在指定位置释放弹道 | 指定位置投掷 |

### 4.4 效果合并优化

当多个效果的目标选取参数完全相同时，系统会合并处理以避免重复搜索：

合并条件（必须全部满足）：
- 目标类型相同
- 过滤条件相同
- 搜索半径相同
- 仅限玩家属性相同
- 脚本检查通过

合并后的结果中，每个被选中的目标都带有一个"效果掩码"，标记它被哪些效果命中。这使得一次搜索的结果可以同时应用于多个效果，大幅提升性能。

## 5 目标选择算法

### 5.1 统一的检查器体系

所有空间搜索算法共享一个统一的检查器体系。检查器是"对每个候选目标进行过滤"的判断函数，它封装了以下通用过滤逻辑：

```
检查器判断流程
│
├── 阵营检查：目标是否符合 CheckType 要求（敌方/友方/队伍/团队）？
│     └── NO → 排除
│
├── 存活检查：目标是否存活？（可选，某些法术可以作用于尸体）
│     └── NO → 排除
│
├── 距离检查：目标是否在有效范围内？
│     └── NO → 排除
│
├── 视线检查：施法者到目标是否有障碍物阻挡？
│     └── NO → 排除（除非法术有忽略视线属性）
│
├── 条件列表检查：是否满足额外条件？（仅条目验证类型）
│     └── NO → 排除
│
└── 全部通过 → 接受为目标
```

不同的空间算法在检查器的基础上增加各自的空间判定：

```
区域检查器 = 通用检查器 + 距离 ≤ 半径
锥形检查器 = 区域检查器 + 角度在锥角范围内
直线检查器 = 区域检查器 + 目标到直线的垂直距离 ≤ 线宽
弹道检查器 = 通用检查器 + 目标到弹道曲线的距离 ≤ 偏差阈值
附近检查器 = 通用检查器 + 距离最近（取第一个匹配）
```

### 5.2 区域选择（Area）

区域选择是最常用的空间搜索算法，适用于所有以圆形范围选取目标的法术。

**核心流程**：

```
1. 确定参考点（根据 ReferenceType）
   - 施法者 → 以施法者位置为中心
   - 目标   → 以显式目标位置为中心
   - 源位置 → 以法术起始坐标为中心
   - 目标位置 → 以法术终止坐标为中心

2. 确定搜索范围
   - 基础半径 + 额外边界余量（避免跨格子遗漏）

3. 执行区域搜索
   - 遍历参考点附近的格子
   - 对每个候选目标执行检查器判定

4. 数量限制
   - 若配置了最大目标数，随机裁剪超出部分
   - 特殊规则（如"最远的 N 个敌人"）按距离排序后截断

5. 脚本拦截
   - 脚本可以修改最终的目标列表（添加/移除/排序）
```

**参考系对搜索的影响**：

| 参考系 | 搜索中心 | 典型法术 |
|--------|---------|---------|
| 施法者 | 施法者位置 | 自身周围 AOE（奉献） |
| 目标 | 显式目标位置 | 目标周围 AOE（群体治疗） |
| 最后 | 上一个效果的命中目标 | 连锁 AOE |
| 源位置 | 法术起始坐标 | 从指定位置释放 |
| 目标位置 | 法术终止坐标 | 地面标记 AOE |

### 5.3 锥形选择（Cone）

锥形选择在圆形搜索的基础上增加了**角度限制**，形成一个扇形区域。

**核心参数**：

```
         锥形区域示意

              锥角
            ┌─────┐
           ╱       ╲
          ╱  有效   ╲
         ╱   区域    ╲
        ╱             ╲
       ╱───────●───────╲  ← 宽度（线宽）
       │       │       │
       │  施法者        │
       └───────●───────┘
              │
           半径

参数：
  - 半径（Radius）：锥形的最大延伸距离
  - 锥角（ConeAngle）：扇形的角度范围（度）
  - 线宽（LineWidth）：锥形底边的厚度（防止扇形过窄）
  - 方向（DirectionType）：锥形的朝向
```

**判定逻辑**：

对每个候选目标：
1. 计算目标到施法者的距离 → 必须在半径范围内
2. 计算目标相对于施法者朝向的角度 → 必须在锥角范围内
3. 执行通用检查器的所有过滤

锥形选择常用于"前方横扫"类技能，如战士的雷霆一击、法师的冰锥术。

### 5.4 链式选择（Chain）

链式选择是一种特殊的选取算法，它不直接搜索区域，而是在**已经选定的初始目标附近**逐步跳跃到额外的目标。

**核心参数**：

| 参数 | 说明 | 典型值 |
|------|------|--------|
| **链式目标数** | 最多跳跃几次 | 3（闪电链）、4（治疗链） |
| **跳跃半径** | 每次跳跃的最大距离 | 10-12.5 码（魔法）、5 码（近战） |
| **伤害衰减** | 每次跳跃的伤害倍率 | 0.7（每次递减 30%） |
| **视线要求** | 跳跃是否需要视线 | 通常需要 |

**链式选择流程**：

```
初始目标（来自显式目标或附近搜索）
      │
      ▼
  跳跃 #1：在初始目标周围 jumpRadius 范围内搜索
      │  条件：不在已有链中的目标、通过视线检查
      │  选距离最近的
      ▼
  跳跃 #2：在上一个目标周围搜索
      │  同上条件
      ▼
  跳跃 #3：...
      │
      ▼
  达到 ChainTargets 上限 → 停止
```

**视线检查的参考点**：
- 默认：从前一个链式目标检查到下一个候选目标的视线
- 特殊属性"从施法者链式"：始终从施法者检查视线
- 特殊属性"从初始目标链式"：始终从初始目标检查视线

**链式伤害衰减**：

链式法术的每次跳跃都会衰减伤害，防止通过增加链数获得过多收益。衰减通过"伤害乘数"实现——第 N 次跳跃的伤害 = 基础伤害 × 衰减系数^(N-1)。

### 5.5 弹道选择（Trajectory）

弹道选择与其他算法不同——它**不选取目标单位**，而是**调整弹道的落点坐标**，使弹道在碰到第一个有效碰撞体时停止。

**核心流程**：

```
1. 前置检查
   - 必须已有弹道数据（仰角、速度）
   - 水平距离不能为零（否则无弹道可言）

2. 计算弹道曲线
   - 根据仰角和速度计算抛物线参数
   - 弹道方程：height = a × distance² + b × distance
   - 其中 a 由仰角和速度决定，b 为弹道斜率

3. 碰撞检测
   - 遍历弹道路径上的所有世界对象
   - 按距离排序（近的先检查）
   - 对每个对象：
     ├── 计算对象到弹道曲线的垂直距离
     ├── 计算对象在弹道高度处的高度差
     └── 若距离和高度差都在阈值内 → 命中！

4. 调整落点
   - 找到第一个命中点后，将目标位置修改到碰撞位置
   - 后续效果（如伤害、AOE）以碰撞位置为中心执行

5. 脚本拦截
   - 脚本可以进一步修改碰撞后的落点坐标
```

弹道选择主要用于"投掷物"类法术，如投石、箭雨。弹道碰撞检测使得投射物可以被建筑物、单位等障碍物阻挡。

### 5.6 直线选择（Line）

直线选择在**从起点到终点的直线路径**上搜索目标，形成一个矩形区域。

**核心参数**：

```
           终点
            ●
           ╱│╲
          ╱ │ ╲
         ╱  │  ╲
        ╱   │   ╲
       ╱ 线 │ 宽 ╲
      ╱     ↓     ╲
     ╱──────────────╲  ← 半径范围
    ●───────●────────●
   起点    中间目标    终点

参数：
  - 起点（Start）：通常为施法者
  - 终点（End）：由 ReferenceType 决定
  - 线宽（LineWidth）：矩形的宽度
  - 半径（Radius）：沿直线的最大延伸距离
```

**判定逻辑**：

对每个候选目标：
1. 计算目标到直线的垂直距离 → 必须在线宽范围内
2. 计算目标在直线上的投影 → 必须在起点-终点范围内
3. 执行通用检查器的所有过滤

直线选择常用于"穿透"类技能，如战士的英勇打击、法师的霜火之箭直线版本。

## 6 目标过滤与验证

### 6.1 过滤链

目标搜索得到的候选列表需要经过多层过滤，确保最终目标都是合法的。

**过滤链的各层级**：

```
候选目标列表
      │
      ├── 阵营过滤：是否符合 CheckType（敌方/友方/队伍/团队）？
      │     └── 不符合 → 排除
      │
      ├── 存活过滤：目标是否存活（可选，复活法术需要尸体）
      │     └── 不符合 → 排除
      │
      ├── 距离过滤：是否在有效范围内（最小/最大距离）
      │     └── 超出范围 → 排除
      │
      ├── 视线过滤：是否有障碍物阻挡（可选）
      │     └── 阻挡 → 排除
      │
      ├── 效果验证：更细粒度的游戏规则检查
      │     ├── 魅惑/控制效果限制
      │     ├── 等级限制
      │     ├── 特殊效果验证（如剥皮需要特定条件）
      │     └── 其他法术属性相关的检查
      │     └── 不通过 → 排除或标记为免疫
      │
      └── 条件列表验证：仅条目验证类型
            ├── 按 Entry 过滤
            ├── 按 Aura 状态过滤
            ├── 按 区域/地图 过滤
            └── 自定义脚本条件
            └── 不通过 → 排除
```

**视线检查的优先级**：

视线（Line of Sight）检查不是总需要的。系统按以下优先级决定是否执行：

1. 法术有"忽略视线"属性 → 跳过视线检查
2. 法术是由触发效果施放的，且触发源有"忽略视线"光环 → 跳过
3. 游戏对象配置了"需要视线" → 必须检查
4. 默认 → 需要检查视线

### 6.2 目标排序与截断

当搜索结果超过预期数量时，系统提供多种排序和截断策略：

| 策略 | 说明 | 典型场景 |
|------|------|---------|
| **随机截断** | 超出部分随机移除 | 大多数 AOE 法术 |
| **距离排序取最远** | 按距离降序，取最远的 N 个 | 特殊 AOE（远者优先） |
| **优先级规则** | 按自定义规则排序后截断 | 脚本自定义的智能选取 |
| **数量上限** | 由法术配置决定最大目标数 | 硬性限制 |

**优先级规则系统**支持最多 31 条自定义排序规则。脚本可以为特定法术注册自定义的排序规则，例如"优先选择血量百分比最低的目标"或"优先选择非 BOSS 目标"。

## 7 脚本拦截机制

### 7.1 三类拦截钩子

目标选择系统提供三类脚本拦截钩子，覆盖不同粒度的修改需求：

| 钩子类型 | 修改粒度 | 调用时机 | 典型用途 |
|---------|---------|---------|---------|
| **区域目标拦截** | 整个目标列表 | 区域搜索完成后 | 移除 BOSS、排序、添加额外目标 |
| **单体目标拦截** | 单一目标指针 | 单体搜索完成后 | 替换为血量最低的队友 |
| **目的地拦截** | 目标位置坐标 | 位置计算完成后 | 调整 AOE 中心点 |

**拦截的执行顺序**：

```
目标搜索完成
      │
      ├── 区域目标拦截
      │     └── 脚本可以：添加/移除/排序目标
      │
      ├── 单体目标拦截
      │     └── 脚本可以：替换目标、设为空（导致法术失败）
      │
      └── 目的地拦截
            └── 脚本可以：修改坐标、偏移位置
```

**阻止默认行为的机制**：区域目标拦截允许脚本完全替换搜索结果。单体目标拦截允许脚本将目标设为空（导致法术以"无效隐式目标"失败）。目的地拦截允许脚本覆盖计算出的位置。

### 7.2 脚本拦截的设计理念

脚本拦截的设计遵循"最小侵入"原则：

- **默认行为完整**：即使没有任何脚本，目标选择也能正确工作
- **拦截点有限**：只在搜索完成后的结果上进行修改，不干预搜索过程本身
- **类型安全**：每种钩子只能修改对应类型的数据（列表/指针/坐标）
- **多脚本协作**：多个脚本可以注册同一钩子，按注册顺序依次执行

## 8 设计决策回顾

### 8.1 关键决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 目标描述方式 | 5 维度参数化 | 可组合、数据驱动、14,000+ 种组合覆盖所有需求 |
| 显式/隐式分离 | 先解析显式再扩展隐式 | 清晰的职责划分，显式来自客户端、隐式来自配置 |
| TargetA + TargetB | 每效果两个目标字段 | 分离主要目标和次要目标，灵活组合 |
| 检查器体系 | 统一基类 + 算法特化 | 通用过滤逻辑复用，各算法只需添加空间判定 |
| 脚本拦截 | 结果修改而非过程干预 | 最小侵入、默认行为完整、类型安全 |

### 8.2 常见陷阱与解决方案

**陷阱 1：跨格子边界遗漏**

区域搜索通常以"格子"为单位遍历世界对象。如果一个目标的搜索范围跨越格子边界，只在参考点所在格子内搜索会遗漏附近格子中的目标。解决方案：搜索半径加上一个"额外格子边界余量"，确保覆盖到相邻格子。

```
正常搜索范围:     ●───────────────●
实际需要搜索范围: ●─────┼──────┼───●
                       │格子边界│
                 需要额外覆盖相邻格子
```

**陷阱 2：链式法术的无限循环**

如果链式法术 A 触发法术 B，法术 B 又触发法术 A，会形成无限递归。解决方案：
- 被触发的法术标记为"触发来源"，不再触发同类效果
- 设置链式深度的硬性上限

**陷阱 3：目标在搜索和命中之间消失**

目标搜索在准备阶段执行，但效果命中可能在数百毫秒后（延迟路径）。期间目标可能死亡、离开范围、或被传送走。解决方案：
- 在效果管线的 Hit 阶段重新验证目标状态（详见效果管线文章的"命中预处理"）
- 不要假设搜索结果在命中时仍然有效

**陷阱 4：显式目标为空但法术需要目标**

玩家可能在没有选中目标时尝试施放需要目标的法术。解决方案：
- 在显式目标解析阶段自动补充（使用当前选中目标、攻击目标、或自身）
- 如果自动补充也失败，以"需要目标"错误拒绝施法

**陷阱 5：引导法术的目标继承**

引导法术的周期性效果需要继承初始的目标列表。如果引导期间目标死亡或离开范围，后续周期不应再对其施加效果。解决方案：
- 引导法术在初始施放时锁定目标列表
- 每个周期开始时检查目标是否仍然有效
- 无效目标从后续周期的列表中移除（但不中断引导本身）

### 8.2 复刻建议

如果你在自己的项目中实现类似的目标选择系统，建议的构建顺序：

**第一阶段：最小可用目标选择**
1. 实现显式目标解析（从客户端请求中提取目标信息）
2. 实现最简单的隐式目标——默认类别（直接使用显式目标）
3. 实现基础的区域选择算法（圆形范围 + 敌方/友方过滤）

**第二阶段：维度化**
4. 定义五维度参数结构
5. 建立静态映射表（将目标类型枚举映射到五维参数）
6. 用维度参数驱动区域选择算法（参考系、验证类型等通过参数传入）

**第三阶段：算法扩展**
7. 添加锥形选择算法
8. 添加直线选择算法
9. 添加链式选择算法

**第四阶段：高级功能**
10. 添加 TargetA + TargetB 组合处理
11. 添加效果合并优化
12. 添加脚本拦截钩子
13. 添加优先级规则系统

**一个关键的早期决策**：你的游戏是否需要投射物？如果所有法术都是即时命中（没有弹道飞行），可以跳过弹道选择算法，大幅降低系统复杂度。类似地，如果不需要链式法术（如闪电链），也可以推迟链式选择的实现。

**性能提示**：区域搜索是性能敏感的热点路径。确保你的世界对象管理使用空间索引（如 Grid/Cell 系统），避免全地图遍历。搜索时只检查参考点附近的格子，并使用额外的边界余量防止跨格子遗漏。

### 8.3 与其他子系统的协作关系

目标选择系统不是孤立工作的，它与法术系统的其他子系统紧密协作：

```
┌─────────────┐  目标列表  ┌─────────────┐  逐目标执行 ┌─────────────┐
│  状态机     │──────────▶│  目标选择   │───────────▶│  效果管线   │
│  (何时执行) │           │  (作用于谁) │             │  (做什么)    │
└─────────────┘           └──────┬──────┘             └─────────────┘
                                 │
                          ┌──────┴──────┐
                          │             │
                    ┌─────┴─────┐ ┌────┴─────┐
                    │  脚本系统  │ │  条件系统 │
                    │  (拦截)   │ │  (过滤)  │
                    └───────────┘ └──────────┘
```

- **状态机**在准备阶段调用目标选择，目标选择完成后状态机才能进入发射阶段
- **效果管线**消费目标选择产出的目标列表，逐目标执行效果
- **脚本系统**通过拦截钩子修改目标选择的结果
- **条件系统**为目标选择提供细粒度的过滤规则
- **视觉系统**使用目标选择的结果决定客户端的法术效果渲染位置
